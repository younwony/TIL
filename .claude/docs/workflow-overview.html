Workflow Overview

Matt Pocock + Track 통합 가이드 · TIL 저장소 워크플로우 한 페이지 시각화
이 페이지는 "Matt Pocock 패턴 전면 재구성" 이후의 전체 워크플로우를 한 페이지에 담는다. 누군가 처음 이 저장소를 열어도 어떤 skill을 언제 부르면 되는지 알 수 있게 만든다.

1. 큰 그림 — 4가지 실패 모드 매핑

AI 코딩 에이전트의 4가지 흔한 실패 모드에 각각 atomic skill을 매핑했다. 큰 워크플로우 skill (/work-plan 등)은 이 atomic skill의 조합 wrapper다 (ADR 0002).

2. 두 가지 모드 — Track 안 vs Track 밖

🚂 Track 안 — 큰 작업 흐름

큰 Jira 작업, 여러 파일 변경, 산출물 보존이 필요할 때.

req.md / 구두 요구사항
    │
    ▼
/work-plan         wrapper
  ├─ grill-with-docs (요구사항 모호 시)
  ├─ to-prd → WORK-SPEC
  ├─ to-issues → FEATURE-CHECKLIST
  └─ Gemini/Codex 크로스체크
    │
    ▼
/work-plan-start   wrapper
  ├─ Solo/Standard/Coordinator 모드
  ├─ tdd 패턴 (각 Phase)
  └─ debugger agent (에러 시 5-phase)
    │
    ▼
/self-review + /qa-scenario + /feature-check
    │
    ▼
/pr + /work-log

산출: .claude/tracks/{id}/0_INDEX.md ~ 8_QA-SCENARIOS.md

🌱 Track 밖 — 즉흥 작업

단일 의도, 1~2파일 수정, atomic skill 단발 호출.

즉흥 의도
    │
    ├─ "이거 필요해?"     → /product-review
    ├─ "용어 정밀화"      → /grill-with-docs
    ├─ "대화 → PRD"       → /to-prd
    ├─ "모듈 deep화"      → /improve-codebase-architecture
    ├─ "throwaway 검증"   → /prototype
    ├─ "이슈 분류"        → /triage
    ├─ "버그 추적"        → debugger agent
    └─ "한 영역 모름"     → /zoom-out

CONTEXT.md/ADR은 자동 갱신됨 (영구 자산). Track 디렉토리 X.

3. Atomic Skill 사용 가이드

atomic = 작고 조합 가능한 단일 책임 skill. wrapper = 여러 atomic을 조합한 큰 워크플로우 skill.

/grill-with-docs atomic

언제: 요구사항 모호, 도메인 용어 충돌, 결정 트리 미해결

한 번에 한 질문씩 grilling. 결정이 풀리는 즉시 CONTEXT.md/ADR 인라인 갱신.

원칙: 코드로 답할 수 있으면 묻지 말고 탐색. ADR은 인색하게 (hard-to-reverse + surprising + real trade-off 3조건).

/triage atomic

언제: 새 Jira 이슈, 백로그 정리, AFK 에이전트가 픽업할 이슈 준비

5-state machine: needs-triage → needs-info | ready-for-agent | ready-for-human | wontfix.

산출: AGENT-BRIEF (interface-level 명세) 또는 .out-of-scope/ 기록.

/improve-codebase-architecture atomic

언제: shallow module 의심, leaky abstraction, untestable 코드

도구: deletion test, Module/Seam/Depth/Leverage/Locality 어휘.

Matt Pocock 권장: 며칠에 한 번씩 코드베이스에 적용.

/to-prd atomic

언제: 토론이 충분히 진행, 결정을 명세로 굳히고 싶음

사용자에게 다시 묻지 않음 — 합성만.

산출: WORK-SPEC.md 양식 (Problem / Solution / User Stories / Implementation / Testing / Out of Scope).

/to-issues atomic

언제: PRD/WORK-SPEC을 작은 단위로 분해

tracer bullet 패턴 — 한 slice = 모든 레이어를 가로지르는 좁은 경로.

HITL/AFK 라벨, 의존성 그래프 포함.

/prototype atomic

언제: state machine/reducer 모호, UI 변형 비교, "이거 동작할까?"

throwaway. 결정만 추출해서 SPEC.md/ADR에 inline 후 prototype 코드 삭제.

5. 실전 시나리오

시나리오 1 — 새 Jira 이슈로 큰 기능 개발

TECH-22386에 "배송 상태 동기화 기능 만들어주세요"라는 한 줄짜리 요구.
1. /product-review               # "왜 만드는가" grilling. Go 판정.
2. /work-plan                    # WORK-SPEC + FEATURE-CHECKLIST 생성
   ├─ (내부) /grill-with-docs   # 용어 정밀화: "배송 상태" = Shipment.status enum 4종
   ├─ (내부) /to-prd            # PRD 합성
   └─ (내부) Gemini/Codex 크로스체크
3. /work-plan-start              # Phase별 구현 (Standard 모드)
4. /self-review                  # 4명 리뷰어 + Gemini/Codex
5. /qa-scenario                  # BDD QA 시나리오
6. /pr                           # GitHub PR
7. /work-log                     # Confluence 작업 로그

시나리오 2 — 짧은 버그 수정

"배송 취소 시 상태가 'cancelled'로 안 바뀜" 버그 리포트.
1. debugger agent (Phase 1 — 재현 loop 만들기 우선)
2. fix + 회귀 테스트
3. /self-review
4. /pr

Track 만들지 않음 (1~2파일 수정).

시나리오 3 — "이 모듈 모양이 잘못된 거 같다"

ShipmentService가 너무 비대하고 테스트가 어려움.
1. /improve-codebase-architecture
   └─ (내부) Explore agent로 코드 탐색
   └─ deletion test 후보 제시
   └─ grilling → "Shipment의 Lifecycle 모듈 추출" 결정
   └─ CONTEXT.md에 "Lifecycle" 용어 추가
   └─ ADR 0004 작성 (옵션)
2. (결정 후) /work-plan 으로 본격 리팩터링 작업 시작

시나리오 4 — 새 화면 UI 변형 비교

배송 상태 페이지의 표시 방식 3가지 후보를 비교하고 싶음.
1. /prototype
   └─ 한 라우트에 3 변형 토글
   └─ 사용자 walkthrough → variant B 선택
   └─ SPEC.md에 결정 inline
   └─ prototype 코드 삭제
2. /work-plan (variant B의 production 구현)

시나리오 5 — Slack 회의 → Jira 이슈

Slack 스레드에 정리된 회의 내용을 Jira 이슈로 분해.
1. /slack-to-jira <Slack URL>    # 1차 변환 (모호한 상태)
2. /triage                        # 5-state machine 분류
   └─ ready-for-agent 후보면 /grill-with-docs로 grilling
   └─ AGENT-BRIEF 코멘트 작성
3. /to-issues                     # vertical slice 분해 (필요 시)

시나리오 6 — 응답이 너무 길어졌을 때

/caveman                    # 토큰 75% 절약 모드 ON
... 작업 ...
"stop caveman"              # 해제 (보안/비가역 명령 시 자동 해제)

시나리오 7 — 처음 보는 코드 영역에서 길 잃음

/zoom-out
   └─ 현재 영역 → 모듈 + 호출자 맵 → CONTEXT.md 어휘로 설명

7. FAQ

Q. /work-plan과 atomic skill 중 무엇을 써야 하나?
변경 파일 3개+ 예상 → /work-plan (Track 만들고 워크플로우 전체). 3개 미만 또는 즉흥 의도 → atomic skill 직접 호출.
Q. /work-plan이 내부에서 atomic skill을 호출한다는데, 어떻게 디버깅하나?
4_PLAN.md에 어떤 atomic skill이 어떤 Step에서 호출되는지 기록됨. 문제 발생 시 해당 atomic skill을 단발 호출하여 재현.
Q. CONTEXT.md / ADR을 언제 갱신하나?
/grill-with-docs, /improve-codebase-architecture 호출 중 자동으로 인라인 갱신된다. 수동 갱신도 가능.
Q. personal/, in-progress/, deprecated/ 버킷의 skill도 호출되나?
호출 가능 (Claude Code의 skill 디스커버리는 깊이 무관). 단 카탈로그/README에 노출 안 되어 외부 사용자에게 보이지 않음.
Q. work-plan-legacy를 다시 쓰고 싶다면?
deprecated/work-plan-legacy/SKILL.md 보존되어 있음. 참조하거나 일시 복귀 가능. 단 새 atomic 패러다임과의 일관성 깨질 수 있음.
Q. Hook이 git add를 차단할 때 어떻게 우회하나?
TIL 저장소 안이면 자동 통과. 그 외 응급 우회: SKIP_CLAUDE_DIR_BLOCK=1 git add ...
Q. 글로벌(~/.claude/)과 TIL의 skill이 다를 수 있나?
기본적으로 /sync-global push|pull로 동기화. TIL 고유 skill (cs-*, pencil-*, weekly-retro 등)은 TIL에만.
