# 셀프 코드 리뷰

PR 생성 전에 현재 브랜치의 변경사항을 자체 리뷰하고 결과를 `SELF-REVIEW.md` 문서로 생성해줘.

## Compare Branch 결정

비교 기준 브랜치를 다음 순서로 결정한다:

1. `git rev-parse --verify main 2>/dev/null` → 존재하면 `main` 사용
2. `git rev-parse --verify master 2>/dev/null` → 존재하면 `master` 사용
3. 위 브랜치가 모두 존재하지 않으면 AskUserQuestion으로 비교 기준 브랜치를 직접 입력받는다
   - "비교 기준 브랜치를 찾을 수 없습니다 (main, master 모두 없음). 비교할 브랜치를 입력해주세요."
4. 결정된 브랜치를 이후 모든 단계에서 `{COMPARE_BRANCH}`로 사용

## 사전 확인

1. 현재 브랜치가 main 또는 base branch가 아닌지 확인
2. main 또는 base branch면 "셀프 리뷰는 작업 브랜치에서 실행해야 합니다."라고 안내하고 중단
3. `where gemini` 명령으로 Gemini CLI 설치 여부 확인
   - 설치됨: Gemini 크로스 리뷰 활성화 (이후 4단계에서 실행)
   - 미설치: "Gemini CLI가 설치되지 않아 Gemini 크로스 리뷰를 건너뜁니다" 안내
4. `where codex` 명령으로 Codex CLI 설치 여부 확인
   - 설치됨: Codex 크로스 리뷰 활성화 (이후 5단계에서 실행)
   - 미설치: "Codex CLI가 설치되지 않아 Codex 크로스 리뷰를 건너뜁니다" 안내

> Gemini와 Codex 모두 비활성화된 경우: "외부 크로스 리뷰 없이 Claude 단독 리뷰로 진행합니다" 안내 후 기존 흐름 유지

## 1단계: 변경사항 수집

다음 명령어들을 **병렬로** 실행하여 정보를 수집:

1. `git branch --show-current` - 현재 브랜치명
2. `git log {COMPARE_BRANCH}..HEAD --oneline` - base branch 이후 커밋 목록
3. `git diff {COMPARE_BRANCH}...HEAD --stat` - 변경된 파일 통계
4. `git diff {COMPARE_BRANCH}...HEAD` - 전체 diff 내용
5. `git status` - 커밋되지 않은 변경사항 확인

## 2단계: 전체 개요 분석

수집한 정보를 바탕으로 다음을 정리:

- **브랜치 목적**: 커밋 메시지들에서 변경 의도 파악
- **주요 변경 파일**: 변경된 파일 목록과 각 파일의 변경 규모 (추가/삭제 라인)
- **커밋 히스토리**: 커밋 메시지 요약

## 3단계: 파일별 심층 리뷰

변경된 각 파일을 Read 도구로 읽고 다음 관점에서 분석:

### 코드 품질
- 네이밍 규칙 준수 여부
- 단일 책임 원칙(SRP) 준수 여부
- 중복 코드 존재 여부
- 매직 넘버/하드코딩된 문자열 사용 여부

### 보안
- 사용자 입력 검증 여부
- SQL 인젝션, XSS 등 OWASP Top 10 취약점
- 민감 정보(비밀번호, API 키 등) 노출 여부

### 성능
- 불필요한 반복문, N+1 쿼리 문제
- 고비용 객체 반복 생성 여부
- 컬렉션 최적화 여부

### 프로젝트 컨벤션
- CLAUDE.md 및 CONVENTION.md에 정의된 규칙 준수 여부
- 커밋 메시지 형식 준수 여부

## 4단계: Gemini 크로스 리뷰

> 사전 확인에서 Gemini CLI가 설치된 경우에만 실행. 미설치 시 이 단계를 건너뛴다.
> 5단계(Codex)와 독립적이므로 **가능한 경우 병렬로 실행**한다.

### 대용량 diff 처리

타임아웃 방지를 위해 diff 크기를 먼저 확인한다:

1. `git diff {COMPARE_BRANCH}...HEAD --stat | tail -1`로 변경 규모를 확인
2. 총 변경 라인이 **1,000줄 이하**인 경우: 전체 diff를 그대로 전달
3. 총 변경 라인이 **1,000줄 초과**인 경우: stat 요약 + 핵심 파일의 diff만 선별 전달
   - `git diff {COMPARE_BRANCH}...HEAD --stat` 결과를 포함
   - 변경 규모가 큰 상위 파일들의 diff만 선택적으로 포함
   - "diff가 커서 주요 파일만 Gemini에 전달되었습니다." 안내

### Gemini 리뷰 실행

다음 명령어로 Gemini에게 코드 리뷰를 요청 (**Bash timeout: 300000ms 필수**):

```bash
# 일반적인 경우 (1,000줄 이하)
git diff {COMPARE_BRANCH}...HEAD | gemini -p "다음 코드 변경사항을 코드 리뷰해줘:

## 리뷰 관점
1. **코드 품질**: 네이밍, SRP, 중복코드, 매직넘버
2. **보안**: OWASP Top 10 취약점, 입력 검증, 민감정보 노출
3. **성능**: N+1 쿼리, 불필요한 반복, 고비용 객체 생성
4. **설계**: SOLID 원칙 준수, 적절한 추상화

## 출력 형식
각 관점별로 발견한 이슈를 심각도(❌ 필수 수정 / ⚠️ 권장 / 💡 참고)와 함께 정리해줘.
한국어로 답변해줘."
```

```bash
# 대용량인 경우 (1,000줄 초과): stat + 주요 파일 diff만 전달
(git diff {COMPARE_BRANCH}...HEAD --stat && echo "---" && git diff {COMPARE_BRANCH}...HEAD -- {주요파일1} {주요파일2} ...) | gemini -p "다음 코드 변경사항을 코드 리뷰해줘:

## 리뷰 관점
1. **코드 품질**: 네이밍, SRP, 중복코드, 매직넘버
2. **보안**: OWASP Top 10 취약점, 입력 검증, 민감정보 노출
3. **성능**: N+1 쿼리, 불필요한 반복, 고비용 객체 생성
4. **설계**: SOLID 원칙 준수, 적절한 추상화

## 출력 형식
각 관점별로 발견한 이슈를 심각도(❌ 필수 수정 / ⚠️ 권장 / 💡 참고)와 함께 정리해줘.
한국어로 답변해줘."
```

- Gemini CLI 실행이 실패하면 (타임아웃, 인증 오류 등) "Gemini 리뷰 실행에 실패했습니다." 안내 후 계속 진행
- Gemini 응답 결과를 변수에 저장하여 7단계에서 SELF-REVIEW.md에 포함

## 5단계: Codex 크로스 리뷰

> 사전 확인에서 Codex CLI가 활성화된 경우에만 실행. 비활성화 시 이 단계를 건너뛴다.
> 4단계(Gemini)와 독립적이므로 **가능한 경우 병렬로 실행**한다.

### 대용량 diff 처리

Gemini 크로스 리뷰(4단계)와 동일한 기준을 적용한다:

1. 총 변경 라인이 **1,000줄 이하**인 경우: 전체 diff를 그대로 전달
2. 총 변경 라인이 **1,000줄 초과**인 경우: stat 요약 + 핵심 파일의 diff만 선별 전달

### Codex 리뷰 실행

다음 명령어로 Codex에게 코드 리뷰를 요청 (**Bash timeout: 300000ms 필수**):

> **참고**: Codex CLI의 `-p` 플래그는 config profile 선택 용도이므로, `codex review --base` 서브커맨드를 사용한다.
> `--base`와 커스텀 프롬프트는 동시 사용이 불가하므로 Codex 기본 리뷰를 수행한다.
> 대용량 diff도 Codex가 자체적으로 처리하므로 별도 분기 불필요.

```bash
codex review --base {COMPARE_BRANCH}
```

- Codex CLI 실행이 실패하면 (타임아웃, 인증 오류 등) "Codex 리뷰 실행에 실패했습니다." 안내 후 계속 진행
- Codex 응답 결과를 변수에 저장하여 7단계에서 SELF-REVIEW.md에 포함

## 6단계: 테스트 커버리지 확인

- 변경된 파일에 대응하는 테스트 파일 존재 여부 확인
- 새로 추가된 기능에 대한 테스트가 충분한지 평가
- 테스트가 부족한 경우 필요한 테스트 케이스 제안

## 7단계: SELF-REVIEW.md 문서 생성

프로젝트 루트에 `SELF-REVIEW.md` 파일을 생성 (이전 내용은 덮어쓰기).
다음 형식으로 작성:

```markdown
# Self Review

> 브랜치: `{브랜치명}` | 리뷰 일시: YYYY-MM-DD HH:MM

## 변경 개요

- **커밋 수**: N개
- **변경 파일 수**: N개
- **추가/삭제**: +N / -N

### 커밋 히스토리

| 커밋 | 메시지 |
|------|--------|
| `abc1234` | 커밋 메시지 |

### 변경 파일

| 파일 | 추가 | 삭제 | 설명 |
|------|------|------|------|
| `path/to/file` | +N | -N | 변경 요약 |

## 리뷰 요약

| 항목 | 상태 | 비고 |
|------|------|------|
| 코드 품질 | ✅ 양호 / ⚠️ 개선 필요 / ❌ 문제 있음 | 한 줄 요약 |
| 보안 | ✅ 양호 / ⚠️ 개선 필요 / ❌ 문제 있음 | 한 줄 요약 |
| 성능 | ✅ 양호 / ⚠️ 개선 필요 / ❌ 문제 있음 | 한 줄 요약 |
| 테스트 | ✅ 충분 / ⚠️ 보완 필요 / ❌ 부족 | 한 줄 요약 |
| 컨벤션 | ✅ 준수 / ⚠️ 일부 미준수 / ❌ 미준수 | 한 줄 요약 |

## 파일별 상세 리뷰

### `path/to/file`

**변경 내용**: 변경 사항 설명
```
