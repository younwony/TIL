# 셀프 코드 리뷰

PR 생성 전에 현재 브랜치의 변경사항을 자체 리뷰하고 결과를 `SELF-REVIEW.md` 문서로 생성해줘.

## Base Branch 결정

다음 순서로 base branch를 결정한다:

1. 프로젝트 CLAUDE.md에서 `PR_BASE_BRANCH` 설정을 찾는다
   - 예: `PR_BASE_BRANCH: main-review`
   - 설정이 있으면 해당 브랜치를 사용
2. 설정이 없으면 다음 순서로 로컬 브랜치 존재 여부를 확인:
   - `git rev-parse --verify main-review 2>/dev/null` → 존재하면 `main-review` 사용
   - `git rev-parse --verify master-review 2>/dev/null` → 존재하면 `master-review` 사용
3. 위 브랜치가 모두 존재하지 않으면 AskUserQuestion으로 base branch를 직접 입력받는다
   - "PR base branch를 찾을 수 없습니다 (main-review, master-review 모두 없음). base branch를 입력해주세요."
4. 결정된 base branch를 이후 모든 단계에서 `{BASE_BRANCH}`로 사용

## 사전 확인

1. 현재 브랜치가 main, main-review가 아닌지 확인
2. main 또는 main-review면 "셀프 리뷰는 작업 브랜치에서 실행해야 합니다."라고 안내하고 중단
3. `where gemini` 명령으로 Gemini CLI 설치 여부 확인
   - 설치됨: Gemini 크로스 리뷰 활성화 (이후 4단계에서 실행)
   - 미설치: "Gemini CLI가 설치되지 않아 Claude 단독 리뷰로 진행합니다" 안내 후 기존 흐름 유지

## 1단계: 변경사항 수집

다음 명령어들을 **병렬로** 실행하여 정보를 수집:

1. `git branch --show-current` - 현재 브랜치명
2. `git log {BASE_BRANCH}..HEAD --oneline` - base branch 이후 커밋 목록
3. `git diff {BASE_BRANCH}...HEAD --stat` - 변경된 파일 통계
4. `git diff {BASE_BRANCH}...HEAD` - 전체 diff 내용
5. `git status` - 커밋되지 않은 변경사항 확인

## 2단계: 전체 개요 분석

수집한 정보를 바탕으로 다음을 정리:

- **브랜치 목적**: 커밋 메시지들에서 변경 의도 파악
- **주요 변경 파일**: 변경된 파일 목록과 각 파일의 변경 규모 (추가/삭제 라인)
- **커밋 히스토리**: 커밋 메시지 요약

## 3단계: 파일별 심층 리뷰

변경된 각 파일을 Read 도구로 읽고 다음 관점에서 분석:

### 코드 품질
- 네이밍 규칙 준수 여부
- 단일 책임 원칙(SRP) 준수 여부
- 중복 코드 존재 여부
- 매직 넘버/하드코딩된 문자열 사용 여부

### 보안
- 사용자 입력 검증 여부
- SQL 인젝션, XSS 등 OWASP Top 10 취약점
- 민감 정보(비밀번호, API 키 등) 노출 여부

### 성능
- 불필요한 반복문, N+1 쿼리 문제
- 고비용 객체 반복 생성 여부
- 컬렉션 최적화 여부

### 프로젝트 컨벤션
- CLAUDE.md 및 CONVENTION.md에 정의된 규칙 준수 여부
- 커밋 메시지 형식 준수 여부

## 4단계: Gemini 크로스 리뷰

> 사전 확인에서 Gemini CLI가 설치된 경우에만 실행. 미설치 시 이 단계를 건너뛴다.

다음 명령어로 Gemini에게 코드 리뷰를 요청:

```bash
git diff main-review...HEAD | gemini -p "다음 코드 변경사항을 코드 리뷰해줘:

## 리뷰 관점
1. **코드 품질**: 네이밍, SRP, 중복코드, 매직넘버
2. **보안**: OWASP Top 10 취약점, 입력 검증, 민감정보 노출
3. **성능**: N+1 쿼리, 불필요한 반복, 고비용 객체 생성
4. **설계**: SOLID 원칙 준수, 적절한 추상화

## 출력 형식
각 관점별로 발견한 이슈를 심각도(❌ 필수 수정 / ⚠️ 권장 / 💡 참고)와 함께 정리해줘.
한국어로 답변해줘."
```

- Gemini CLI 실행이 실패하면 (타임아웃, 인증 오류 등) "Gemini 리뷰 실행에 실패했습니다. Claude 단독 리뷰로 진행합니다." 안내 후 계속 진행
- Gemini 응답 결과를 변수에 저장하여 6단계에서 SELF-REVIEW.md에 포함

## 5단계: 테스트 커버리지 확인

- 변경된 파일에 대응하는 테스트 파일 존재 여부 확인
- 새로 추가된 기능에 대한 테스트가 충분한지 평가
- 테스트가 부족한 경우 필요한 테스트 케이스 제안

## 6단계: SELF-REVIEW.md 문서 생성

프로젝트 루트에 `SELF-REVIEW.md` 파일을 생성 (이전 내용은 덮어쓰기).
다음 형식으로 작성:

```markdown
# Self Review

> 브랜치: `{브랜치명}` | 리뷰 일시: YYYY-MM-DD HH:MM

## 변경 개요

- **커밋 수**: N개
- **변경 파일 수**: N개
- **추가/삭제**: +N / -N

### 커밋 히스토리

| 커밋 | 메시지 |
|------|--------|
| `abc1234` | 커밋 메시지 |

### 변경 파일

| 파일 | 추가 | 삭제 | 설명 |
|------|------|------|------|
| `path/to/file` | +N | -N | 변경 요약 |

## 리뷰 요약

| 항목 | 상태 | 비고 |
|------|------|------|
| 코드 품질 | ✅ 양호 / ⚠️ 개선 필요 / ❌ 문제 있음 | 한 줄 요약 |
| 보안 | ✅ 양호 / ⚠️ 개선 필요 / ❌ 문제 있음 | 한 줄 요약 |
| 성능 | ✅ 양호 / ⚠️ 개선 필요 / ❌ 문제 있음 | 한 줄 요약 |
| 테스트 | ✅ 충분 / ⚠️ 보완 필요 / ❌ 부족 | 한 줄 요약 |
| 컨벤션 | ✅ 준수 / ⚠️ 일부 미준수 / ❌ 미준수 | 한 줄 요약 |

## 파일별 상세 리뷰

### `path/to/file`

**변경 내용**: 변경 사항 설명

**리뷰 의견**:
- {구체적인 리뷰 의견}

## Gemini 크로스 리뷰

> Reviewed by: Gemini (via Gemini CLI)
> Gemini CLI가 설치되지 않았거나 실행에 실패한 경우 이 섹션은 생략한다.

{Gemini CLI 리뷰 결과를 그대로 포함}

## 개선 제안

우선순위별로 정리:

### 필수 수정 (❌)
{PR 생성 전 반드시 수정해야 할 사항}

### 권장 수정 (⚠️)
{수정하면 좋은 사항}

### 참고 (💡)
{참고할 만한 사항}

## 결론

{Claude 분석과 Gemini 리뷰(있는 경우) 양쪽 결과를 종합한 최종 리뷰 의견 요약. PR 생성 가능 여부 판단}
```

## 7단계: 다음 액션 선택

문서 생성 후 AskUserQuestion으로 다음 액션을 선택:

- **PR 생성 진행**: `/pr` 스킬을 이어서 실행
- **개선 사항 직접 수정**: 리뷰에서 발견된 문제를 바로 수정
- **종료**: 셀프 리뷰 종료
