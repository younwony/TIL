# Product Review

기능 구현 전에 "왜 만드는가"를 구조적으로 검증하는 스킬이다.
코드를 한 줄도 쓰기 전에, 이 기능이 정말 필요한지, 더 간단한 방법은 없는지, 리스크는 무엇인지를 점검한다.

## 왜 이 스킬이 필요한가

개발에서 가장 비싼 낭비는 "필요 없는 기능을 잘 만드는 것"이다.
AI 도구가 강력해질수록 구현 비용이 낮아지면서, "일단 만들어보자"는 유혹이 커진다.
하지만 만든 뒤의 유지보수, 테스트, 문서화 비용은 여전히 크다.
이 스킬은 구현 전에 한 발 물러서서 "정말 이게 최선인가?"를 묻는 구조화된 프로세스를 제공한다.

---

## 워크플로우

### Step 1: 맥락 수집

현재 작업 맥락을 파악한다. 아래 소스를 순서대로 탐색한다:

1. **사용자가 직접 설명한 기능** (대화 컨텍스트)
2. **WORK-SPEC.md** (`.claude/tracks/*/1_WORK-SPEC.md` 또는 `.claude/docs/1_WORK-SPEC.md`)
3. **req.md** (요구사항 파일)
4. **현재 브랜치 변경사항** (`git log`, `git diff`)
5. **관련 Jira 이슈** (브랜치명에서 이슈 키 추출)

수집된 맥락을 1~2문장으로 요약하여 확인한다:
> "현재 검토 대상: {기능 설명}"

### Step 2: 6가지 검증 질문

수집된 맥락을 기반으로 아래 6가지 질문에 답변을 작성한다.
각 질문에 대해 **답변 초안**을 제시하고, 사용자에게 확인/수정을 요청한다.

#### Q1. 핵심 문제 (What problem does this solve?)
> 이 기능이 해결하려는 **구체적인 문제**는 무엇인가?
> "~하면 좋겠다"가 아니라 "~때문에 불편하다/불가능하다"로 표현할 수 있는가?

#### Q2. 대상 사용자 (Who has this problem?)
> 이 문제를 **실제로 겪는 사용자**는 누구인가?
> 구체적인 페르소나나 사용 시나리오가 있는가?

#### Q3. 현재 해결책 (How is this solved today?)
> 이 기능 없이 **현재 어떻게 이 문제를 해결**하고 있는가?
> 수동 프로세스? 다른 도구? 아니면 참고 있는 중?

#### Q4. 최소 해결책 (Is there a simpler solution?)
> 이 기능을 만들지 않고 **더 간단하게 해결**하는 방법은 없는가?
> 설정 변경? 기존 도구 활용? 프로세스 변경?

#### Q5. 성공 측정 (How will you measure success?)
> 이 기능이 성공했는지 **어떻게 판단**할 것인가?
> 사용률? 오류 감소? 시간 절약? 측정 가능한 지표가 있는가?

#### Q6. 유지보수 비용 (What's the ongoing cost?)
> 구현 후 **지속적으로 드는 비용**은?
> 테스트 유지, 버전 호환, 문서 업데이트, 사용자 지원 등

### Step 3: 4가지 확장 분석

검증 질문 답변을 기반으로 4가지 관점에서 추가 분석한다:

#### 문제 재정의 (Reframe)
> 문제를 다르게 정의하면 더 나은 해결책이 보이는가?
> - 원래 문제: {Q1의 답}
> - 재정의: {다른 관점에서 본 문제}
> - 이 관점에서의 해결책: {대안}

#### 범위 축소 (Minimize)
> MVP(최소 기능 제품)로 줄이면 어떻게 되는가?
> - 전체 기능: {현재 계획}
> - MVP: {핵심만 남긴 버전}
> - 빠진 것의 가치: {MVP에서 뺀 것이 정말 필요한지}

#### 대안 탐색 (Alternatives)
> 완전히 다른 접근법은 없는가?
> - 대안 1: {설명}
> - 대안 2: {설명}
> - 각 대안의 장단점

#### 위험 분석 (Risks)
> 최악의 시나리오는 무엇인가?
> - 구현 실패 시: {영향}
> - 잘못 구현 시: {영향}
> - 구현 후 사용되지 않을 시: {매몰 비용}
> - 의존성/기술 부채 위험: {평가}

### Step 4: Go/No-Go 판단

모든 분석을 종합하여 판단을 제시한다:

| 판단 | 조건 |
|------|------|
| **Go** | 문제가 명확하고, 대안이 없으며, 유지보수 비용이 합리적 |
| **Go (축소)** | 가치는 있으나, MVP로 범위를 줄여서 진행 권장 |
| **Hold** | 추가 정보가 필요하거나, 우선순위 재검토 필요 |
| **No-Go** | 문제가 불명확하거나, 더 간단한 대안이 존재 |

```markdown
## 제품 검증 결과

### 요약
- 검토 대상: {기능명}
- 판단: **{Go / Go(축소) / Hold / No-Go}**
- 근거: {1~2문장}

### 검증 질문 요약
| # | 질문 | 답변 요약 | 확신도 |
|---|------|----------|--------|
| 1 | 핵심 문제 | ... | 높음/중간/낮음 |
| 2 | 대상 사용자 | ... | ... |
| 3 | 현재 해결책 | ... | ... |
| 4 | 최소 해결책 | ... | ... |
| 5 | 성공 측정 | ... | ... |
| 6 | 유지보수 비용 | ... | ... |

### 확장 분석
- 문제 재정의: {핵심}
- MVP 제안: {핵심}
- 대안: {핵심}
- 위험: {핵심}

### 권고
{구체적 다음 단계}
```

### Step 5: 후속 연결

판단 결과에 따라 다음 워크플로우를 안내한다:

| 판단 | 다음 단계 |
|------|----------|
| Go | `/work-plan`으로 작업 명세서 생성 |
| Go (축소) | MVP 범위로 조정 후 `/work-plan` |
| Hold | 추가 조사 사항 목록 제시 |
| No-Go | 대안 실행 계획 또는 종료 |

---

## 주의사항

- 이 스킬은 **구현 전 검증** 도구이다. 이미 구현이 진행 중인 경우에는 `/self-review`가 더 적합하다.
- 검증 질문의 답변은 **초안**이다. 사용자의 도메인 지식이 반드시 필요하므로 대화형으로 진행한다.
- "No-Go" 판단이 나왔다고 해서 무조건 중단해야 하는 것은 아니다. 최종 결정은 사용자의 몫이다.
- 학습 목적(TIL)의 작업은 "유지보수 비용"보다 "학습 가치"에 더 높은 가중치를 부여한다.
