# Weekly Retro

한 주간의 작업을 자동으로 분석하여 "무엇을 했고, 무엇을 배웠고, 무엇을 개선할 수 있는지"를 정리하는 스킬이다.
git 히스토리를 기반으로 객관적 데이터를 수집하고, 회고 질문에 대한 답변 초안을 제시한다.

## 왜 이 스킬이 필요한가

TIL(Today I Learned) 프로젝트에서 학습 효과를 극대화하려면 정기적인 회고가 필수다.
하지만 수동 회고는 기억에 의존하기 때문에 빠뜨리는 것이 많고, 패턴을 발견하기 어렵다.
git 히스토리는 거짓말을 하지 않는다 — 이를 기반으로 객관적 회고를 자동화한다.

---

## 워크플로우

### Step 1: 기간 설정

기본값은 **최근 1주일**이다. 사용자가 기간을 지정하면 그에 맞춘다.

```bash
# 기본: 최근 1주일
git log --since="1 week ago" --format="%H|%ai|%s" --no-merges

# 이번 달 전체
git log --since="1 month ago" --format="%H|%ai|%s" --no-merges
```

커밋이 없으면 "해당 기간에 커밋이 없습니다"로 종료한다.

### Step 2: 커밋 분석

커밋을 다음 카테고리로 자동 분류한다:

| 카테고리 | 판별 기준 | 예시 |
|----------|----------|------|
| CS 학습 | `cs/` 경로 변경 | CS 문서 작성, 다이어그램 추가 |
| 스터디 | `study/` 경로 변경 | 프로젝트 실습, 책 정리 |
| 도구 개선 | `.claude/` 경로 변경 | 스킬, 커맨드, 설정 |
| 인프라 | `CLAUDE.md`, `.gitignore` 등 | 컨벤션, 설정 변경 |
| 기타 | 위에 해당하지 않는 것 | README, 기타 문서 |

```bash
# 카테고리별 파일 변경 통계
git log --since="1 week ago" --no-merges --name-only --format="" | sort | uniq -c | sort -rn
```

### Step 3: 통계 수집

```bash
# 일별 커밋 수
git log --since="1 week ago" --format="%ad" --date=short --no-merges | sort | uniq -c

# 변경된 파일 수
git log --since="1 week ago" --no-merges --shortstat

# 가장 많이 수정된 파일 Top 10
git log --since="1 week ago" --no-merges --name-only --format="" | sort | uniq -c | sort -rn | head -10
```

### Step 4: 회고 질문 & 인사이트

수집된 데이터를 기반으로 다음 질문에 답변 초안을 작성한다:

#### 성과 (What went well)
- 이번 주 가장 가치 있었던 작업은? (커밋 수, 파일 변경량 기준 + 내용 분석)
- 새로 배운 것 중 가장 인상적인 것은?

#### 개선 (What could be improved)
- 반복된 패턴이나 비효율은? (같은 파일 반복 수정, 되돌린 변경 등)
- 커밋 메시지에서 발견되는 패턴은? (fix가 많다면 → 초기 설계 개선 필요)

#### 다음 주 (What to try next)
- 이번 주 시작했지만 완료하지 못한 것은?
- 다음 주에 시도해볼 새로운 접근법은?

#### 학습 깊이 분석 (TIL 프로젝트 특화)
- CS 카테고리별 문서 수: 어떤 분야에 집중했는가?
- 난이도 분포: 기초/중급/심화 비율
- 연속 학습 일수 (streak)

### Step 5: 회고 문서 생성

분석 결과를 마크다운 문서로 생성한다.

**저장 위치**: `cs/retro/weekly-retro-{YYYY-MM-DD}.md`

```markdown
# 주간 회고 ({시작일} ~ {종료일})

> 이 문서는 {시작일}~{종료일} 기간의 작업을 자동 분석한 주간 회고입니다.

## 한 주 요약

- 총 커밋: {N}건
- 변경 파일: {N}개
- 주요 작업 영역: {카테고리 Top 3}

## 일별 활동

| 날짜 | 커밋 수 | 주요 작업 |
|------|---------|----------|
| ... | ... | ... |

## 카테고리별 분석

### CS 학습
- {내용}

### 스터디
- {내용}

### 도구 개선
- {내용}

## 회고

### 잘한 것
- {자동 생성 초안}

### 개선할 것
- {자동 생성 초안}

### 다음 주 계획
- {자동 생성 초안}

## 학습 통계

- CS 문서 작성: {N}건
- 연속 학습: {N}일째
```

### Step 6: 사용자 피드백

생성된 회고 문서를 보여주고, 사용자가 수정하거나 보완할 내용이 있는지 확인한다.
확정되면 `git add`까지 수행한다.

---

## 주의사항

- `.claude/` 하위 변경은 "도구 개선" 카테고리로 분류하되, git add 대상에서는 제외한다.
- 커밋 메시지가 영어/한글 혼용일 수 있으므로 양쪽 모두 파싱한다.
- 회고 질문의 답변은 **초안**이다. 사용자가 직접 수정하는 것을 권장한다.
