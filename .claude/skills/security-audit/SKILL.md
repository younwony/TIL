# Security Audit (STRIDE + OWASP)

STRIDE 위협 모델링과 OWASP Top 10을 결합하여 변경된 코드의 보안을 체계적으로 감사하는 스킬이다.
기존 `review-security` 에이전트가 OWASP 위주였다면, 이 스킬은 STRIDE로 위협을 분류하고 OWASP로 구체적 취약점을 점검한다.

## 왜 이 스킬이 필요한가

OWASP Top 10은 "무엇이 취약한가"를 알려주지만, "어떤 위협이 존재하는가"는 설명하지 않는다.
STRIDE는 공격자 관점에서 위협을 분류하여 누락 없이 보안을 점검할 수 있게 한다.
두 프레임워크를 결합하면 위협 식별(STRIDE)과 취약점 점검(OWASP)을 모두 커버할 수 있다.

---

## STRIDE 프레임워크

| 위협 | 설명 | 점검 포인트 |
|------|------|------------|
| **S**poofing (위장) | 다른 사용자/시스템으로 위장 | 인증 메커니즘, 세션 관리, API 키 검증 |
| **T**ampering (변조) | 데이터나 코드를 무단 수정 | 입력 검증, 서명 확인, CSRF 토큰 |
| **R**epudiation (부인) | 행위를 부인할 수 있는 상태 | 감사 로그, 트랜잭션 기록, 타임스탬프 |
| **I**nformation Disclosure (정보 노출) | 민감 정보가 노출 | 에러 메시지, 로그, API 응답, 설정 파일 |
| **D**enial of Service (서비스 거부) | 서비스를 사용 불가로 만듦 | 리소스 제한, 타임아웃, 무한 루프 |
| **E**levation of Privilege (권한 상승) | 더 높은 권한을 획득 | 인가 검증, 역할 확인, 관리자 기능 보호 |

## 워크플로우

### Step 1: 변경 파일 식별

```bash
git diff --cached --name-only
git diff --name-only
git ls-files --others --exclude-standard
```

비코드 파일(`.md`, `.yml`, 이미지 등)은 제외한다.
변경 파일이 없으면 종료한다.

### Step 2: 보안 컨텍스트 파악

변경된 파일의 성격을 파악하여 관련 STRIDE 항목에 가중치를 부여한다:

| 파일 유형 | 주요 STRIDE 항목 |
|----------|----------------|
| 컨트롤러/API 엔드포인트 | S, T, I, E |
| 인증/인가 코드 | S, E |
| 데이터 액세스/JPA | T, I |
| 설정 파일 | I, E |
| 파일 I/O | T, I |
| 외부 API 호출 | S, T, D |

### Step 3: STRIDE 분석

변경된 각 파일을 Read 도구로 읽고, STRIDE 6개 항목을 순서대로 점검한다.

#### S - Spoofing (위장)
- [ ] 인증 없이 접근 가능한 엔드포인트가 있는가?
- [ ] 세션/토큰 검증이 우회 가능한가?
- [ ] API 키/시크릿이 하드코딩되어 있는가?

#### T - Tampering (변조)
- [ ] 사용자 입력이 검증 없이 사용되는가? (SQL, XSS, Command Injection)
- [ ] 요청 파라미터로 다른 사용자의 데이터에 접근 가능한가? (IDOR)
- [ ] 파일 경로가 사용자 입력에서 오는가? (Path Traversal)

#### R - Repudiation (부인)
- [ ] 중요 작업에 감사 로그가 남는가?
- [ ] 로그에 충분한 컨텍스트(누가, 언제, 무엇을)가 포함되는가?
- [ ] 로그 조작이 가능한가?

#### I - Information Disclosure (정보 노출)
- [ ] 에러 응답에 스택 트레이스나 내부 정보가 포함되는가?
- [ ] 로그에 민감 정보(비밀번호, 토큰, 개인정보)가 출력되는가?
- [ ] API 응답에 불필요한 필드가 포함되는가?

#### D - Denial of Service (서비스 거부)
- [ ] 페이징 없는 대량 조회가 가능한가?
- [ ] 파일 업로드 크기 제한이 있는가?
- [ ] 무한 루프나 재귀 가능성이 있는가?
- [ ] 타임아웃이 설정되어 있는가?

#### E - Elevation of Privilege (권한 상승)
- [ ] 관리자 기능에 역할 검증이 있는가?
- [ ] 메서드 레벨 인가(@PreAuthorize 등)가 적용되어 있는가?
- [ ] 다른 사용자의 리소스에 접근할 수 없는가?

### Step 4: OWASP Top 10 크로스체크

STRIDE 분석 결과를 OWASP Top 10과 매핑하여 누락된 항목이 없는지 확인한다.

| OWASP | 대응 STRIDE | 추가 점검 |
|-------|-----------|----------|
| A01 접근 제어 취약 | S, E | 수평/수직 권한 상승 |
| A02 암호화 실패 | I | 평문 저장, 약한 해시 |
| A03 인젝션 | T | SQL, XSS, LDAP, OS Command |
| A04 불안전한 설계 | 전체 | 위협 모델링 자체의 부재 |
| A05 보안 설정 오류 | I, E | 기본 계정, 불필요한 기능 활성화 |
| A06 취약한 컴포넌트 | 전체 | 의존성 버전 확인 |
| A07 인증 실패 | S | 브루트포스, 약한 비밀번호 정책 |
| A08 무결성 실패 | T | 서명 미검증, 안전하지 않은 역직렬화 |
| A09 로깅 실패 | R | 감사 로그 부재 |
| A10 SSRF | T, D | 서버 측 요청 위조 |

### Step 5: 결과 보고

```markdown
## 보안 감사 결과

### 요약
- 분석 파일: {N}개
- 발견 사항: {N}건
- 신뢰도 점수: {score}/10

### STRIDE 분석 결과

| 위협 | 발견 | 심각도 | 파일 | 설명 |
|------|------|--------|------|------|
| Spoofing | ... | 높음/중간/낮음 | ... | ... |
| Tampering | ... | ... | ... | ... |
| ... | ... | ... | ... | ... |

### OWASP 크로스체크
- A01~A10 중 해당 사항: {목록}

### 수정 권고
1. [심각] {설명} → {수정 방법}
2. [경고] {설명} → {수정 방법}
3. ...
```

심각도가 "높음"인 항목이 있으면 수정 코드를 함께 제안한다.

## 신뢰도 점수 기준

| 점수 | 의미 |
|------|------|
| 9-10 | 보안 위협 없음, 베스트 프랙티스 준수 |
| 7-8 | 경미한 개선 사항 있음, 프로덕션 배포 가능 |
| 5-6 | 보안 개선 필요, 수정 후 배포 권장 |
| 3-4 | 심각한 취약점 존재, 수정 필수 |
| 1-2 | 치명적 취약점, 즉시 수정 필요 |

## 주의사항

- 이 스킬은 **정적 분석**이다. 런타임 취약점(예: 환경 설정 문제)은 탐지하지 못한다.
- 프레임워크 보안 설정(Spring Security 등)은 설정 파일이 변경 대상에 포함될 때만 점검한다.
- 의존성 취약점(A06)은 `build.gradle`/`pom.xml` 변경 시에만 확인한다.
- 기존 `review-security` 에이전트와 병행 사용 가능하다. 이 스킬은 STRIDE 프레임워크를 추가한 상위 호환이다.
