# CS 문서 작성 템플릿

## 난이도 체계

난이도는 3개로 고정하지 않고 유연하게 세분화할 수 있습니다.

**핵심 원칙:**
1. **카테고리 정의가 최우선** - 각 카테고리는 "~란?" 정의 문서로 시작
2. **선수 지식 기반 순서** - 선수 지식이 없는 것 → 있는 것 순서

| 레벨 | 설명 | 표기 예시 |
|------|------|----------|
| `[1] 정의/기초` | 카테고리 정의, 선수 지식 없음 | `> \`[1] 정의\` · 선수 지식 없음` |
| `[2] 입문` | 기본 개념, 간단한 선수 지식 | `> \`[2] 입문\` · 선수 지식: 없음` |
| `[3] 중급` | 실무 핵심, 여러 기초 지식 | `> \`[3] 중급\` · 선수 지식: [문서명](./path.md)` |
| `[4] 심화` | 복잡한 개념, 다중 선수 지식 | `> \`[4] 심화\` · 선수 지식: [문서1](./path.md), [문서2](./path.md)` |
| `[5+]` | 필요 시 추가 가능 | - |

## 다이어그램 스펙 작성법

다이어그램 생성은 `cs-diagram-generator` 에이전트에 위임합니다. 기획 단계에서 스펙만 정의합니다.

### 방식 선택 기준

> **Mermaid 우선 정책**: 속도 최적화를 위해 **Mermaid를 기본 방식**으로 사용합니다.
> SVG 직접 생성은 Mermaid로 표현이 불가능한 경우에만 사용합니다.

| 방식 | 적합한 경우 | 속도 |
|------|-----------|------|
| **Mermaid** (기본) | 플로우차트, 시퀀스, 상태, ER, 클래스, 마인드맵, Git 그래프, 대부분의 다이어그램 | 빠름 (~30초) |
| **SVG 직접** (예외) | Mermaid로 표현 불가한 경우만: 복잡한 네트워크 토폴로지, 패킷/프레임 구조, 정밀 좌표 레이아웃 | 느림 (~90초) |

**Mermaid로 대체 가능한 경우:**

| 기존 SVG 직접 | Mermaid 대체 방식 |
|--------------|------------------|
| 계층 구조도 | `graph TD` (트리 구조) |
| 비교표/매트릭스 | `quadrantChart` 또는 마크다운 테이블 |
| 생태계 지도 | `mindmap` 또는 `graph LR` 클러스터 |
| 아키텍처 다이어그램 | `graph TD` + `subgraph` |

### 스펙 정의 형식

기획 단계 결과물에서 아래 형식으로 작성합니다:

```
다이어그램 목록:
1. {파일명} ({유형}, "{설명}", {방식: Mermaid / SVG 직접})
2. {파일명} ({유형}, "{설명}", {방식: Mermaid / SVG 직접})
```

### 마크다운 삽입법

문서 본문에 미리 참조를 삽입합니다 (에이전트가 파일을 실제로 생성):

```markdown
![{설명}](./images/{name}.svg)
```

### 트러블슈팅 품질 기준

| 조건 | 설명 |
|------|------|
| 재현 가능 | 같은 환경에서 같은 문제를 재현할 수 있어야 함 |
| 구체적 수치 | 에러 메시지, 로그, 메트릭 등 구체적 데이터 포함 |
| 원인 분석 | "왜 발생했는지" 기술적 분석 필수 |
| 다중 해결책 | 가능하다면 여러 해결 방법 제시 |
| 예방 조치 | 재발 방지를 위한 모니터링, 설정 가이드 |
