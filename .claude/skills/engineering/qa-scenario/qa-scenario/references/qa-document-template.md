# QA 시나리오 문서 템플릿 레퍼런스

Phase 5에서 `QA-SCENARIOS.md`를 생성할 때 참조하는 상세 템플릿.

## 완전한 템플릿

```markdown

## 환경
- 브랜치: {branch_name}
- 비교 기준: {compare_branch} (보통 main 또는 develop)
- 테스트 URL: http://localhost:{port}
- 생성 일시: {YYYY-MM-DD HH:mm}
- 변경 파일: {N}개
- 생성 도구: qa-scenario skill

## 변경 요약

{변경된 기능을 1~2줄로 요약. 기술적 세부사항보다 비즈니스 관점의 변경 내용에 초점}

예시:
- "브랜드 누락 관리 페이지 신규 개발. 목록 조회, 상세 보기, 상태 변경 기능 포함."
- "상품 등록 폼에 카테고리 자동완성 기능 추가 및 이미지 업로드 UX 개선."

## 영향도 매트릭스

| 변경 파일 | 카테고리 | 직접 영향 | 간접 영향 | 위험도 |
|----------|---------|----------|----------|--------|
| {파일명} | Controller | {직접 영향 받는 기능} | {간접 영향 받는 기능} | 🔴/🟡/🟢 |

위험도 기준:
- 🔴 높음: 핵심 기능(CRUD, 인증, 결제) 변경, 다수 사용자 영향
- 🟡 중간: 부가 기능 변경, 특정 조건에서만 영향
- 🟢 낮음: UI/텍스트 변경, 설정 변경

## 유저 플로우 다이어그램

변경사항이 영향을 주는 전체 사용자 동선을 SVG 이미지로 표현.
전체 시스템이 아닌, 변경과 관련된 흐름만 포함한다.

`cs-diagram-generator` 에이전트를 사용하여 `qa-images/user-flow.svg`로 생성한다.

스타일 규칙:
- 시작점: 파랑 (#e1f5fe)
- 성공 경로: 초록 (#c8e6c9)
- 실패 경로: 빨강 (#ffcdd2)
- 분기점: 마름모 형태

문서에는 이미지 참조로 포함:
```markdown
![유저 플로우](qa-images/user-flow.svg)
```

## 시나리오 작성 팁

### Given 조건 잘 작성하기

나쁜 예: "페이지에 접속한 상태"
좋은 예: "관리자 권한으로 로그인하고, 상품 데이터가 20건 이상 존재하는 상태에서"

Given은 재현 가능해야 한다. 필요한 데이터, 권한, 사전 작업을 구체적으로 명시한다.

### When 액션 구체적으로

나쁜 예: "저장 버튼 클릭"
좋은 예: "상품명을 '테스트 상품'으로 입력하고, 카테고리를 '전자기기'로 선택한 후, 저장 버튼을 클릭하면"

### Then 결과 검증 가능하게

나쁜 예: "정상적으로 동작한다"
좋은 예: "POST /api/items가 201으로 응답하고, 성공 토스트 '저장되었습니다'가 표시되며, 목록 그리드가 자동 갱신된다"

### 검증 코드 작성 원칙

1. DOM 셀렉터는 최대한 구체적으로 (`.ag-row` > `div`)
2. `console.assert`로 즉시 확인 가능하게
3. 비동기 동작은 setTimeout 또는 MutationObserver 고려
4. 네트워크 검증은 URL 패턴 + HTTP 상태코드 + 응답 구조

| 변경 유형 | 필수 다이어그램 | 선택 다이어그램 |
|----------|--------------|--------------|
| 페이지 신규 생성 | flowchart | journey |
| 기존 페이지 수정 | flowchart | sequence (API 변경 시) |
| API 추가/변경 | sequence | flowchart |
| 상태 전이 로직 | stateDiagram | flowchart |
| CRUD 전체 | flowchart + sequence | journey |
| UI만 변경 (CSS/JS) | flowchart | - |
| 설정만 변경 | - (시나리오만) | - |

## 재사용 시나리오

이 문서는 다음 상황에서 재사용된다:

1. **browser-debug 연계**: QA-SCENARIOS.md를 그대로 Chrome 자동화 입력으로 사용
2. **회귀 테스트**: 동일 기능 수정 시 기존 시나리오 기반 재실행
3. **PR 리뷰**: 변경 영향도 파악 참고 자료
4. **인수 테스트**: QA 팀/PO가 기능 검증 체크리스트로 활용
5. **히스토리**: 버그 발견/수정 이력 추적
