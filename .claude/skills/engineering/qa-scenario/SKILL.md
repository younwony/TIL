# QA Scenario Generator

현재 브랜치의 변경사항을 분석하여 체계적인 QA 시나리오 문서를 생성한다.
반드시 아래 Phase를 순서대로 수행한다.

---

## Phase 1: 변경사항 수집 (자동)

### 1-1. Git 변경사항 수집

```bash
# staged + unstaged 변경 파일 목록
git diff --cached --name-status
git diff --name-status
git status --short

# 변경 내용 상세
git diff --cached
git diff

# 비교 브랜치 대비 전체 변경 (PR용)
git log --oneline main..HEAD
git diff main...HEAD --stat
```

### 1-2. 변경 파일 분류

수집된 파일을 아래 카테고리로 분류한다:

| 카테고리 | 파일 패턴 | 분석 포인트 |
|---------|----------|------------|
| **Controller** | `*Controller.java` | URL 엔드포인트, HTTP 메서드, 요청/응답 DTO |
| **Service** | `*Service.java`, `*ServiceImpl.java` | 비즈니스 로직 변경, 트랜잭션 경계 |
| **Repository** | `*Repository.java`, `*Mapper.xml` | 쿼리 변경, 데이터 모델 |
| **Entity/DTO** | `*Entity.java`, `*Dto.java`, `*Request.java`, `*Response.java` | 필드 변경, 유효성 검증 |
| **View** | `*.jsp`, `*.html`, `*.ftl`, `*.mustache` | UI 렌더링, 레이아웃, 폼 구조 |
| **JavaScript** | `*.js`, `*.ts`, `*.vue`, `*.jsx`, `*.tsx` | 이벤트 핸들러, API 호출, DOM 조작 |
| **CSS** | `*.css`, `*.scss`, `*.less` | 스타일링, 반응형, 레이아웃 |
| **설정** | `application.*`, `*.xml`, `build.gradle`, `pom.xml` | 설정 변경, 빈 등록, 의존성 |
| **테스트** | `*Test.java`, `*Spec.java`, `*.test.*` | 테스트 커버리지 변경 |
| **기타** | 위에 해당하지 않는 파일 | 간접 영향 분석 |

## Phase 1.5: 유저 인터뷰 (대화)

> 코드 변경만 보면 진입점·사용자 역할·비즈니스 목적을 알 수 없다.
> 유저와 직접 대화하여 분석 컨텍스트를 확정한 뒤 Phase 2~3으로 진입한다.
> **같은 Track 내에서는 이전 답변을 자동 재사용하여 반복 질문을 피한다.**

### 1.5-0. 적용 조건

- 변경 파일 **3개 이상**일 때만 인터뷰 진행
- 변경 파일 1~2개면 이 Phase 전체 스킵 (소규모 변경은 코드만으로 충분)

### 1.5-1. 캐시 확인 (자동 재사용)

활성 Track이 있으면 `{DOC_DIR}/QA-CONTEXT.md` 파일을 먼저 확인한다.

다음 조건 중 하나라도 해당하면 캐시 사용 안 함 → 1.5-2부터 인터뷰 진행:
- 캐시 파일이 존재하지 않음
- `$ARGUMENTS`에 `refresh`, `갱신`, `다시`, `재인터뷰` 키워드 포함
- 사용자가 직전 대화에서 "인터뷰 다시 받자" 등 명시적 갱신 요청을 했음

캐시를 사용하는 경우:
1. 파일을 읽어 4개 항목(진입점·플로우, 사용자·데이터, 목적, 리스크)을 추출
2. 사용자에게 한 줄 요약을 표시:
   ```
   📋 이 Track의 QA 인터뷰 컨텍스트를 재사용합니다 (마지막 갱신: {mtime}).
   - 진입점/플로우: {요약}
   - 사용자/데이터: {요약}
   - 목적/배경: {요약}
   - 알려진 리스크: {요약}
   → 갱신이 필요하면 `/qa-scenario refresh` 또는 "인터뷰 다시 받자"로 호출해주세요.
   ```
3. **인터뷰 단계 전체 스킵하고 Phase 2로 진행**

### 1.5-2. 추정값 생성

Phase 1에서 수집한 정보를 바탕으로 4개 항목의 추정값을 미리 만든다:

- **진입점 + 핵심 유저 플로우**: Controller의 `@RequestMapping` URL과 View 파일을 조합하여 추정 (예: "관리자가 `/admin/items/list`에서 상품 등록 후 노출 토글 진행")
- **사용자 역할/권한 + 사전 데이터**: 코드의 `@PreAuthorize`, Spring Security 설정, DTO 필드를 보고 추정 (예: "슈퍼관리자, 등록된 상품 5개 이상 필요")
- **비즈니스 목적/배경**: 커밋 메시지에서 추출 (예: "노출 정책 변경: 미노출 상품의 검색 결과 제외")
- **알려진 리스크/엣지 케이스**: 변경 규모·테스트 누락·외부 연동 여부에서 추정 (예: "동시 토글 시 캐시 무효화 누락 우려")

추정 근거가 빈약한 항목은 추정값 대신 `"(추정 불가 — 직접 입력 부탁드립니다)"` 로 표시한다.

### 1.5-3. 인터뷰 진행 (AskUserQuestion 4문항 일괄 호출)

`AskUserQuestion`으로 4개 질문을 **한 번에** 호출한다 (한 번의 인터럽트로 종료).

각 질문의 옵션 구조:

| 옵션 | 라벨 | 동작 |
|------|------|------|
| 1 | `{추정값}으로 진행` (Recommended) | 추정값 채택 |
| 2 | `잘 모르겠음 / 스킵` | 해당 차원 가중치 낮춤 |
| Other (자동) | (사용자 자유 입력) | 입력값 채택 |

질문 4개:
1. **진입점·핵심 유저 플로우**: "이 변경의 진입 화면과 핵심 유저 플로우가 다음과 같나요? (추정: {추정값})"
2. **사용자 역할·사전 데이터**: "테스트 대상 사용자와 사전 데이터 조건이 다음과 같나요? (추정: {추정값})"
3. **비즈니스 목적·배경**: "이번 변경의 비즈니스 목적이 다음과 같나요? (추정: {추정값})"
4. **알려진 리스크·엣지 케이스**: "특별히 신경 써야 할 리스크/엣지 케이스가 다음과 같나요? (추정: {추정값})"

### 1.5-4. 응답 정규화

각 항목 최종값:
- 옵션 1 선택 → 추정값 사용
- Other 입력 → 사용자 입력 사용
- 옵션 2 (스킵) → `"(스킵됨)"` 표시, Phase 2~3에서 해당 차원 가중치 낮춤

### 1.5-5. 컨텍스트 저장 (Track 캐시)

활성 Track이 있는 경우에만 `{DOC_DIR}/QA-CONTEXT.md`를 생성/덮어쓴다:

```markdown
# QA Interview Context

- 브랜치: {branch}
- 마지막 갱신: {YYYY-MM-DD HH:mm}
- 변경 파일 수: {N}

## 진입점 + 핵심 유저 플로우
{값}

## 사용자 역할/권한 + 사전 데이터
{값}

## 비즈니스 목적/배경
{값}

## 알려진 리스크/엣지 케이스
{값}
```

- 활성 Track이 없는 경우: 파일 저장 안 함, 메모리에만 보관하여 Phase 2~3에서 사용
- 파일은 `{DOC_DIR}` 하위이므로 `git add` 하지 않음 (CLAUDE.md `.claude/` 정책 준수)

### 1.5-6. Phase 2~3에 컨텍스트 주입

후속 Phase에 다음과 같이 반영:

- **Phase 2 (영향 범위 분석)**: Explore 에이전트에게 "사용자가 명시한 진입점 `{값}`을 기준으로 호출 체인 추적" 지시
- **Phase 3 (시나리오 초안)**:
  - Given 조건에 사용자 역할·사전 데이터를 그대로 반영
  - P0 선정 시 비즈니스 목적·리스크 항목을 가중치로 사용
  - "알려진 리스크"로 명시된 케이스는 P0/P1으로 자동 승급

## Phase 2: 영향 범위 분석 (Plan 모드)

**EnterPlanMode**로 Plan 모드에 진입한 후, Explore 에이전트를 **`model: "haiku"`** 로 디스패치하여 변경 코드의 영향 범위를 "very thorough"로 분석한다.

Agent 도구 파라미터:
- `subagent_type`: `"Explore"`
- `model`: `"haiku"`
- `thoroughness`: `"very thorough"`

### 2-1. 코드 흐름 추적

변경된 코드가 영향을 미치는 전체 흐름을 추적한다:

- **진입점 파악**: Controller의 URL 매핑 → 어떤 페이지/API에서 호출되는가
- **호출 체인**: Controller → Service → Repository → DB 순서로 추적
- **프론트엔드 연결**: JSP/HTML에서 해당 API를 호출하는 JS 코드, 폼, 버튼
- **사용자 인터랙션**: onclick, submit, jQuery handler, AJAX 호출 등
- **모달/팝업**: 동적으로 열리는 모달, confirm 다이얼로그
- **외부 연동**: 파일 업로드/다운로드, 외부 API 호출
- **상태 변화**: 데이터 상태 전이 (예: 대기→처리→완료)

### 2-2. 영향도 매트릭스

분석 결과를 영향도 매트릭스로 정리한다:

```markdown
| 변경 파일 | 직접 영향 | 간접 영향 | 위험도 |
|----------|----------|----------|--------|
| ItemController.java | /api/items CRUD | 목록 페이지, 상세 페이지 | 🔴 높음 |
| ItemService.java | 저장 로직 | 검증, 이벤트 발행 | 🟡 중간 |
| item-list.js | 그리드 렌더링 | 필터, 페이지네이션 | 🟡 중간 |
```

위험도 기준:
- 🔴 **높음**: 핵심 기능(CRUD, 인증, 결제) 또는 다수 사용자 영향
- 🟡 **중간**: 부가 기능 또는 특정 조건에서만 영향
- 🟢 **낮음**: UI 변경, 텍스트 수정, 설정 변경

### 2-3. 테스트 범위 결정

영향도 매트릭스를 기반으로 테스트 범위를 결정한다:

- 🔴 높음 → **P0 시나리오** (반드시 테스트)
- 🟡 중간 → **P1 시나리오** (주요 기능)
- 🟢 낮음 → **P2 시나리오** (부가 기능)

## Phase 3: QA 시나리오 초안 작성 (Plan 모드 유지)

### 3-1. 다이어그램 생성 (SVG 이미지)

변경 유형에 따라 적절한 다이어그램을 선택한다:

| 변경 유형 | 필수 다이어그램 | 선택 다이어그램 |
|----------|--------------|--------------|
| 페이지 신규/수정 | flowchart (유저 플로우) | journey (사용자 경험) |
| API 추가/변경 | sequenceDiagram | flowchart |
| 상태 전이 로직 | stateDiagram | flowchart |
| CRUD 전체 | flowchart + sequenceDiagram | journey |
| UI만 변경 | flowchart | - |

**flowchart 스타일 규칙:**
- 성공 경로: `style node fill:#c8e6c9` (초록)
- 실패 경로: `style node fill:#ffcdd2` (빨강)
- 시작점: `style node fill:#e1f5fe` (파랑)
- 분기점: 마름모 `{}` 사용

**SVG 생성 절차:**

다이어그램은 외부 파일이 아닌 **본문에 인라인 `<svg>`로 임베드**한다 (자체 완결 HTML 원칙).

1. `html-doc/references/components.html`의 플로우 다이어그램 스니펫을 참조한다
2. 변경 유형에 맞게 SVG를 커스터마이즈하여 `<figure><svg>...</svg><figcaption>...</figcaption></figure>` 형태로 본문에 직접 삽입한다:
   - 유저 플로우 → 플로우 다이어그램 SVG
   - API 시퀀스 → 시퀀스 표현 SVG (해당 시)
   - 영향도 분포 → 막대 차트 SVG (선택)
3. SVG fill/stroke에 CSS 변수(`var(--accent)`, `var(--muted)` 등)를 사용하여 다크모드 자동 대응한다

> 인라인 SVG로 표현이 너무 복잡한 경우에만 Mermaid 코드 블록을 `<pre>` 안에 포함한다.

### 3-2. 시나리오 작성 형식

모든 시나리오는 BDD Given-When-Then 형식으로 작성한다:

```markdown
#### S01. {기능명}
- **우선순위**: P0
- **테스트 URL**: {URL}
- **시나리오**:
  - **Given**: {사전 조건 - 로그인 상태, 데이터 존재 여부 등}
  - **When**: {사용자 액션 - 클릭, 입력, 선택 등}
  - **Then**: {기대 결과 - 화면 변화, API 응답, 데이터 변경 등}
- **검증 방법**: {screenshot / javascript / network / console}
- **검증 코드**:
  ```javascript
  // 자동화 검증에 사용할 구체적인 코드
  ```
- **결과**: ⬜ 미실행
```

### 3-2b. 제미나이 검증 체크리스트 (각 시나리오 1:1 매칭)

> Claude Code(시나리오 작성) → 개발자(상태 조작) → 제미나이(화면/DOM 검증) 핑퐁 워크플로우의 검증 단계용.
> 제미나이는 마우스/키보드를 못 쓰고 **현재 띄워진 화면의 정적인 상태**만 분석한다.
> 따라서 "행위"가 아닌 "상태"를 묻는 체크리스트를 각 BDD 시나리오와 1:1로 함께 작성한다.

#### 작성 원칙 (행위형 ❌ → 상태형 ✅)

| 원칙 | ❌ 행위형 (제미나이가 못 함) | ✅ 상태형 (제미나이가 가능) |
|------|---------------------------|---------------------------|
| 명사화 | "로그인 버튼 클릭해줘" | "'로그인' 버튼이 활성화(Enabled) 상태인가?" |
| 구체적 타겟 | "버튼이 있어?" | "'로그인' 버튼이 존재하는가?" |
| 텍스트 인용 | "환영 메시지 보여?" | "'환영합니다' 텍스트가 정상적으로 노출되어 있는가?" |
| 측정 가능한 표현 | "잘 보여?" | "에러 메시지가 붉은색 텍스트(예: #dc3545)로 노출되어 있는가?" |
| 단답 가능 | "어떻게 보여?" | "체크박스에 체크 표시가 되어 있는가? (Y/N)" |

#### 체크리스트 카테고리

3개 카테고리로 분류한다 (필수 2 + 선택 1):

| 카테고리 | 필수 여부 | 점검 대상 |
|---------|---------|----------|
| **UI 및 렌더링 체크** | 필수 | 요소 존재 여부, 텍스트 정확성, 개수 |
| **상태 및 속성 체크** | 필수 | Enabled/Disabled, placeholder, value, checked, aria-* |
| **레이아웃/시각적 체크** | 선택 | 정렬, 색상, 여백, 반응형 — 시각 검증이 필요한 경우만 |

#### 상태 분리 (인터랙션이 있는 시나리오)

BDD의 `When`이 사용자 인터랙션을 포함하면 체크리스트를 두 묶음으로 작성한다:

- **A. 초기 상태**: 페이지 로드 직후, `When` 수행 전
- **B. 인터랙션 후 상태**: 개발자가 `When`을 직접 수행한 직후

`When`이 없는 단순 페이지 로드 시나리오는 **A. 초기 상태만** 작성한다.

#### 시나리오별 작성 템플릿

각 BDD 시나리오 아래에 다음 markdown 블록을 작성한다. **사용자가 그대로 복사하여 제미나이에게 던질 수 있는 형태**여야 한다:

```markdown
##### S01 제미나이 검증 체크리스트

**[화면/기능명]**: {기능명}

### A. 초기 상태 (페이지 로드 직후)

**UI 및 렌더링 체크**
- [ ] '{텍스트}' 제목 텍스트가 정상적으로 노출되어 있는가?
- [ ] {필드명} 입력 필드(Input)가 {N}개 존재하는가?
- [ ] '{링크명}' 링크 텍스트가 존재하는가?

**상태 및 속성 체크**
- [ ] '{버튼명}' 버튼이 {Enabled/Disabled} 상태인가?
- [ ] {필드명} placeholder가 "{값}"으로 되어 있는가?

**레이아웃/시각적 체크** (선택)
- [ ] {영역}이 화면 {중앙/좌측/우측}에 정렬되어 있는가?

### B. 인터랙션 후 상태 ({When 1줄 요약})

> 개발자가 {조작 내용}을 수행한 직후 검증.

**상태 변화 체크**
- [ ] '{버튼명}' 버튼이 {Enabled/Disabled}로 전환되었는가?
- [ ] {요소}의 텍스트가 "{변경 후 값}"으로 바뀌었는가?
- [ ] (에러 케이스) 에러 메시지 "{메시지}"가 노출되어 있는가?
```

#### 자체 검증 — 체크리스트 품질 게이트

체크리스트를 작성한 후 다음을 자체 점검한다. 한 항목이라도 위반하면 다시 작성:

- [ ] 모든 항목이 **물음표(?)로 끝나는 단답형 질문**인가?
- [ ] **동사형 명령**("클릭해줘", "스크롤해줘", "확인해줘")이 한 개도 없는가?
- [ ] **요소 이름이 따옴표로 인용**되어 있는가? (예: `'로그인' 버튼`)
- [ ] **모호한 표현**("적절히", "예쁘게", "자연스럽게")이 없는가?
- [ ] **DOM 또는 스크린샷으로 확인 가능**한 항목만 포함되어 있는가? (API 응답, DB 상태, console 로그는 BDD 본문의 `검증 코드`로 처리하고 체크리스트에서는 제외)

### 3-3. 시나리오 분류 기준

| 우선순위 | ID 범위 | 대상 | 예시 |
|---------|---------|------|------|
| **P0** | S01~S04 | 핵심 기능 — 이것이 실패하면 배포 불가 | 페이지 로드, 메인 CRUD, 핵심 API |
| **P1** | S05~S09 | 주요 기능 — 사용성에 직접 영향 | UI 인터랙션, 필터, 정렬, 페이지네이션 |
| **P2** | S10~ | 부가 기능 — 엣지 케이스 | 엑셀, 빈 데이터, 외부 연동, 날짜 범위 |

### 3-4. 검증 방법 가이드

| 검증 유형 | 사용 상황 | 자동화 도구 |
|----------|----------|-----------|
| **screenshot** | UI 레이아웃, 스타일 확인 | Chrome 캡처 |
| **javascript** | DOM 상태, 변수 값, 그리드 데이터 | `javascript_tool` |
| **network** | API 호출 URL, 상태코드, 응답 데이터 | `read_network_requests` |
| **console** | JS 에러, 경고, 디버그 로그 | `read_console_messages` |

검증 코드는 `browser-debug` 스킬의 Chrome 자동화에서 바로 사용할 수 있도록 구체적으로 작성한다.

## Phase 5: 8_QA-SCENARIOS.html 생성

QA 시나리오 문서는 `html-doc` 스킬 규칙을 따라 자체 완결 HTML로 작성한다.
`references/qa-document-template.html`을 사용한다. BDD 시나리오는 `<details>`로,
영향도 매트릭스는 비교표로, 우선순위는 상태 배지로 표현한다.
산출 HTML 문서에는 html-doc 스킬의 시각화 가이드에 따라 핵심 흐름·영향도를 인라인 SVG 다이어그램으로 1개 이상 포함한다.

`{DOC_DIR}/8_QA-SCENARIOS.html` 파일을 생성한다. (`{DOC_DIR}`은 글로벌 CLAUDE.md의 워크플로우 문서 저장 경로 규칙을 따른다.)
(`references/qa-document-template.html` 참조)

### 문서 구조

`qa-document-template.html`을 skeleton으로 복사하고 아래 섹션을 채운다:

- **환경**: 브랜치명, 비교 기준, 테스트 URL, 생성 일시, 변경 파일 수
- **변경 요약**: 변경된 기능 1~2줄 요약
- **제미나이 핑퐁 워크플로우 안내** `<aside>` (문서 상단 1회): Claude Code(시나리오) → 개발자(조작) → 제미나이(검증) 흐름을 인라인 `<svg>` 다이어그램과 함께 설명. "각 시나리오의 markdown 블록을 복사하여 제미나이에게 전달" 안내 포함
- **영향도 매트릭스**: `<table>`로 변경 파일별 직접/간접 영향, 위험도 표시
- **핵심 흐름 다이어그램**: 인라인 `<svg>`로 유저 플로우 또는 API 시퀀스 표현 (1개 이상 필수)
- **시나리오 목록**: P0/P1/P2 각 `<details>` + 배지
  - 각 시나리오 내부 구조:
    1. BDD Given-When-Then 본문 (`<dl>` 또는 `<table>`)
    2. 검증 방법·검증 코드 (`<pre><code class="language-javascript">`)
    3. **"📋 제미나이 검증 체크리스트 (복사하여 전달)"** 안내 헤더
    4. raw markdown 체크리스트 블록 (`<pre><code class="language-markdown">...</code></pre>`)
       - `When`이 있는 시나리오는 **A. 초기 상태**, **B. 인터랙션 후 상태** 두 블록 모두 포함
       - `When`이 없는 시나리오는 A 블록만 포함
- **결과 요약**: QA 실행 후 업데이트 예정 표
- **발견된 버그**: `<table>`
- **미검증 항목**: `<ul>`

### 결과 상태 아이콘

| 상태 | 아이콘 | 설명 |
|------|--------|------|
| 미실행 | ⬜ | 아직 테스트하지 않음 |
| 실행중 | 🔄 | 현재 테스트 진행중 |
| 통과 | ✅ | 기대 결과와 일치 |
| 실패 | ❌ | 기대 결과와 불일치 |
| 수정후통과 | 🔧 | 버그 발견 → 수정 → 재검증 통과 |
| 차단됨 | ⚠️ | 선행 조건 미충족으로 테스트 불가 |

`.claude/tracks/` 디렉토리에서 status가 `in_progress`인 활성 Track을 탐색한다.

## 활용 시나리오

### 독립 사용
`/qa-scenario` → QA 시나리오 문서 생성 → 팀 공유 / PR 첨부

### browser-debug 연계
`/qa-scenario` → QA 시나리오 생성 → `/browser-debug` → Chrome 자동화 실행

### PR 리뷰 첨부
`/qa-scenario` → QA 시나리오 생성 → PR description에 포함

### 제미나이 핑퐁 검증 (Chrome 브라우저 AI)

`/qa-scenario` → BDD + 제미나이 체크리스트 동시 생성 →

1. 사용자가 로컬에서 대상 페이지를 띄운다
2. 시나리오의 **"A. 초기 상태"** markdown 블록을 복사하여 제미나이에게 전달
3. 제미나이가 화면을 분석해 항목별로 `[Pass]` / `[Fail]` 리포트
4. (인터랙션이 있는 경우) 사용자가 직접 입력/클릭 등을 수행 → **"B. 인터랙션 후 상태"** 블록을 재전달하여 상태 변화 검증
5. `[Fail]` 항목은 개발자가 수정 후 같은 블록으로 재요청

> 자동화는 `browser-debug` 스킬(Chrome MCP), 인간+제미나이 정적 검증은 이 체크리스트 — 둘은 보완 관계다.

## 주의사항

- Plan 모드에서 충분히 분석한 후 시나리오를 제안한다 (성급하게 문서 생성 X)
- 변경 코드를 직접 읽어 영향 범위를 파악한다 (파일명만 보고 추측 X)
- 검증 코드는 `browser-debug` 스킬에서 바로 실행 가능하도록 구체적으로 작성한다
- 시나리오가 너무 많으면 (15개 초과) P0/P1에 집중하고 P2는 간략하게 작성한다
- 테스트 데이터가 필요한 시나리오는 Given 조건에 명시한다

## 관련 스킬

- `browser-debug`: QA 시나리오를 Chrome 자동화로 실행
- `test-coverage-check`: 단위/통합 테스트 커버리지 분석
- `self-review`: PR 전 코드 자체 리뷰
