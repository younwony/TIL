# Zoom Out

> 이 코드 영역을 잘 모릅니다. 한 단계 추상화를 올려서, 관련 모듈과 호출자(caller) 맵을 **도메인 용어**로 설명해주세요.

## 동작

사용자가 특정 파일·메서드·디렉토리를 가리키며 "잘 모르겠다"고 할 때, 다음을 수행한다:

1. **`.claude/CONTEXT.md` 도메인 용어집을 먼저 읽는다** (있으면). 거기 정의된 어휘로 설명한다. 일반 프로그래밍 용어가 아닌 프로젝트 고유 어휘를 우선 사용.

2. **모듈 맵**: 이 영역과 직접 관련된 모듈 5~10개를 나열. 각 모듈은:
   - 한 줄 책임
   - 인터페이스(주요 entry point만)
   - CONTEXT.md 어휘로 표현된 역할

3. **호출자 맵**: 누가 이 영역을 호출하는가. 어떤 도메인 시점에 호출되는가.

4. **호출 대상 맵**: 이 영역이 무엇을 호출하는가. 외부(DB, API)는 별도 표시.

5. **한 단계 위 추상화**: 이 영역을 한 줄로 표현하면? "이건 X가 Y할 때 Z를 하는 곳."

## 출력 양식

```markdown
## Zoom Out: {대상}

### 한 단계 위 추상화
{이 영역은 X가 Y할 때 Z를 하는 곳입니다 — CONTEXT.md 어휘 사용}

### 관련 모듈 (도메인 용어로)
| 모듈 | 책임 | 도메인 역할 |
|------|------|-----------|
| ... | ... | ... |

### 호출자 (이 영역을 누가 부르나)
- {Caller A}: {언제, 어떤 시나리오}
- ...

### 호출 대상 (이 영역이 무엇을 부르나)
- {Callee X} (in-process)
- {Callee Y} (DB/외부 API)

### 다음 행동 제안
- 더 깊이 보고 싶다면: {구체적 파일/메서드}
- 위로 더 올라가고 싶다면: {부모 영역}
```

## 규칙

- **CONTEXT.md 어휘 우선**. 없으면 코드의 식별자에서 추론.
- **추측하지 말고, 모르면 모른다고 말한다**. "아마도 X일 것"보다 "확신할 수 없습니다, 직접 확인하세요" 가 낫다.
- **5~10개 모듈로 제한**. 더 많으면 한 번 더 zoom out.
- 파일 경로는 보조. 도메인 역할이 메인.
- 사용자가 "더 깊이"라고 하면 zoom in, "더 위로"라고 하면 한 번 더 zoom out.
