# Feature Check

FEATURE-CHECKLIST.md의 각 항목을 **코드 레벨에서** 자동 검증하여 구현 완료 여부를 확인한다.
브라우저 QA(`/browser-debug`) 전에 "코드가 실제로 존재하는가?"를 먼저 확인하는 단계이다.

## 왜 이 스킬인가?

- 기능 체크리스트가 있어도 실제 코드에 반영됐는지 확인하려면 파일을 하나하나 열어봐야 한다
- 이 스킬은 각 체크리스트 항목을 코드/API/DB/설정 관점에서 자동 검증한다
- 통과 항목은 `[x]`로 체크하고, 미구현 항목은 이슈와 함께 보고한다
- 브라우저 QA 전에 코드 레벨 게이트를 통과시켜, QA에서 "코드가 없어서 실패"하는 낭비를 방지

## 실행 방법

```
/feature-check                    # 활성 Track의 *_FEATURE-CHECKLIST.md 자동 탐색
/feature-check path/to/checklist  # 지정 경로의 체크리스트 사용
```

## 워크플로우 위치

```
/work-plan → /work-plan-start → /test-coverage-check → /feature-check → /self-review → /qa-scenario → /browser-debug → /pr
```

`/test-coverage-check`(테스트 커버리지) 이후, `/self-review`(코드 리뷰) 이전에 실행한다.
테스트가 통과해도 "요구된 기능이 실제로 구현되었는가?"는 별도 확인이 필요하기 때문이다.

## 작업 절차

### 1. 체크리스트 탐색

`$ARGUMENTS`가 있으면 해당 경로, 없으면 활성 Track에서 `*_FEATURE-CHECKLIST.md`를 패턴매칭으로 찾는다.

```
탐색 순서:
1. $ARGUMENTS로 전달된 경로
2. .claude/tracks/{active-track}/*_FEATURE-CHECKLIST.md
3. 없으면 사용자에게 경로 질문
```

### 2. 체크리스트 파싱

체크리스트에서 모든 `- [ ]` 항목을 추출한다. 이미 `- [x]`인 항목은 건너뛴다.
카테고리(## 헤더)별로 그룹화한다.

### 3. 항목별 검증 전략 결정

각 미체크 항목을 읽고, **검증 방법을 자동 분류**한다:

| 검증 유형 | 판별 기준 | 검증 방법 |
|----------|----------|----------|
| **코드 존재** | "~가 표시된다", "~버튼이 있다", "~함수가 동작한다" | Grep/Read로 해당 코드/컴포넌트 존재 확인 |
| **API 동작** | "~API 호출", "~요청 시", "~응답" | 컨트롤러 메서드 + 서비스 로직 존재 확인, 컴파일 검증 |
| **DB/쿼리** | "~테이블", "~컬럼", "~쿼리" | MyBatis XML/Entity 필드 존재 확인 |
| **설정** | "~설정", "~옵션", "~프로퍼티" | application.yml/JSP/JS 설정 확인 |
| **로직** | "~시 ~한다", "~검증", "~차단" | 서비스/JS에서 해당 조건분기 코드 존재 확인 |
| **수동 확인** | 위 어디에도 해당 안 됨 | `[?]`로 마킹, 사용자에게 위임 |

검증은 **코드가 존재하는가**를 확인하는 것이지, **런타임 동작**을 확인하는 것이 아니다.
런타임 동작은 `/browser-debug`에서 확인한다.

### 4. 검증 실행

Explore 에이전트를 활용하여 병렬로 검증한다.

**병렬 전략:**
- 카테고리가 3개 이상이면 카테고리별로 Explore 에이전트 디스패치
- 카테고리가 2개 이하면 Main에서 직접 검증

**에이전트 프롬프트 템플릿:**
```
다음 기능 체크리스트 항목들을 코드에서 검증해줘. 각 항목에 대해:
1. 관련 코드가 존재하는지 Grep/Read로 확인
2. 존재하면 파일경로:라인번호 근거를 제시
3. 존재하지 않으면 "미구현"으로 표시

프로젝트: {프로젝트 루트 경로}
카테고리: {카테고리명}
항목:
{체크리스트 항목 목록}
```

**검증 판정 기준:**
- **PASS**: 관련 코드가 명확히 존재 (함수, 컴포넌트, 쿼리, 설정 등)
- **FAIL**: 코드가 없거나, 관련 로직이 누락
- **MANUAL**: 코드 레벨로 판단 불가 (UI 레이아웃, 사용자 경험 등)

### 5. 체크리스트 업데이트

검증 결과에 따라 체크리스트 파일을 업데이트한다:

| 결과 | 마킹 | 설명 |
|------|------|------|
| PASS | `- [x]` | 코드 존재 확인됨 |
| FAIL | `- [ ]` (유지) | 미구현 — 이슈 목록에 추가 |
| MANUAL | `- [ ]` (유지) | 수동 확인 필요 — 별도 표시 |

### 6. 결과 보고

```markdown
## Feature Check 결과

### 요약
- 검증 대상: {N}개
- PASS (코드 확인): {N}개
- FAIL (미구현): {N}개
- MANUAL (수동 확인 필요): {N}개
- 이미 체크됨 (스킵): {N}개

### PASS 항목
| 카테고리 | 항목 | 근거 |
|---------|------|------|
| {카테고리} | {항목 텍스트} | `파일:라인` |

### FAIL 항목 (미구현)
| 카테고리 | 항목 | 사유 |
|---------|------|------|
| {카테고리} | {항목 텍스트} | {코드 미발견 상세} |

### MANUAL 항목 (수동 확인 필요)
| 카테고리 | 항목 | 사유 |
|---------|------|------|
| {카테고리} | {항목 텍스트} | {판단 불가 사유} |

### 다음 단계
- FAIL 항목이 있으면 → 구현 후 `/feature-check` 재실행
- 모두 PASS면 → `/self-review`로 코드 리뷰 진행
```

### 7. 컴파일 검증 (Java 프로젝트)

Java/Kotlin 프로젝트인 경우 마지막에 `./gradlew compileJava` (또는 `mvn compile`)를 실행하여 컴파일 검증을 수행한다.
컴파일 실패 시 FAIL 항목에 추가한다.

---

## 주의사항

- **코드 존재 확인만**: 런타임 동작 검증은 하지 않는다. 그건 `/browser-debug`의 역할.
- **체크리스트 원본 보존**: FAIL/MANUAL 항목은 `- [ ]`를 유지한다. PASS만 `- [x]`로 변경.
- **V2 코드 수정 금지**: V3 패키지 내에서만 검증. V2 파일은 참조만.
- **false positive 주의**: 코드가 존재해도 로직이 불완전할 수 있다. 의심스러우면 MANUAL로 분류.
- **이전 결과 존중**: 이미 `[x]`인 항목은 재검증하지 않는다. 강제 재검증은 `--force` 옵션.
