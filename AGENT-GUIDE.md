# 실무 Sub-Agent 가이드

Claude Code에서 사용할 수 있는 **커스텀 Sub-Agent** 가이드입니다.
각 에이전트는 독립 컨텍스트에서 전용 시스템 프롬프트와 도구 접근 권한을 가지고 자율적으로 작업을 수행합니다.

## Sub-Agent vs Command 차이점

```
┌─────────────────────────────────────────────────────┐
│  Command (슬래시 명령어)                              │
│                                                       │
│  - 메인 대화 컨텍스트에서 실행                         │
│  - /pr, /self-review 등                              │
│  - 사용자가 직접 호출                                  │
│  - 메인 컨텍스트를 소비함                              │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  Sub-Agent (에이전트)                                 │
│                                                       │
│  - 독립 컨텍스트 윈도우에서 실행                       │
│  - Claude가 작업 유형에 따라 자동 위임                 │
│  - 명시적 요청도 가능                                  │
│  - 메인 컨텍스트를 보존함 (결과 요약만 반환)            │
│  - 모델, 도구, 권한을 독립적으로 설정                  │
│  - 백그라운드 실행 가능                                │
└─────────────────────────────────────────────────────┘
```

## 에이전트 목록

| # | 이름 | 모델 | 핵심 가치 |
|---|------|------|----------|
| 1 | `test-generator` | Sonnet | 코드 분석 → 테스트 작성 → 실행 → 실패 시 자동 수정 |
| 2 | `code-refactor` | Sonnet | 코드 스멜 분석 → 개선안 제시 → 승인 후 적용 |
| 3 | `debugger` | Sonnet | 스택 트레이스 분석 → 원인 추적 → 수정 제안/적용 |
| 5 | `jira-updater` | Haiku | 이슈 감지 → 상태 전환 → 코멘트 추가 |


## 위임 방식

에이전트에게 작업을 위임하는 방법은 **자동 위임**과 **수동 위임** 두 가지가 있습니다.

### 자동 위임

Claude가 사용자 요청의 의도와 에이전트의 `description`을 비교하여 자동으로 적절한 에이전트에 위임합니다.

```
사용자: "UserService의 테스트를 만들어줘"
→ Claude가 test-generator 에이전트의 description과 매칭
→ 자동으로 에이전트에 위임
→ 에이전트가 독립 컨텍스트에서 작업 수행
→ 결과 요약만 메인 대화로 반환
```

### 수동 위임

에이전트 이름을 직접 지정하여 명시적으로 위임할 수 있습니다.
자동 위임이 되지 않거나, 특정 에이전트를 지정하고 싶을 때 사용합니다.

```
"test-generator 에이전트로 UserService 테스트를 생성해줘"
"debugger 에이전트를 사용해서 이 에러를 분석해줘"
"code-refactor 에이전트한테 변경 파일 리팩토링 맡겨줘"
```

### 실행 모드

#### 포그라운드 (기본)

메인 대화를 블로킹하고 에이전트가 작업을 완료할 때까지 대기합니다.
사용자에게 권한 확인이나 질문을 전달할 수 있습니다.

#### 백그라운드

에이전트가 백그라운드에서 작업하는 동안 메인 대화를 계속할 수 있습니다.

```
"테스트 생성을 백그라운드에서 진행해줘"
```

- 실행 전 필요한 도구 권한을 미리 승인받음
- 실행 중 Ctrl+B로 포그라운드 ↔ 백그라운드 전환 가능
- 작업 완료 시 결과를 메인 대화로 반환

#### 이어서 작업 (Resume)

에이전트가 완료한 이후에도 이전 컨텍스트를 유지한 채 작업을 이어갈 수 있습니다.

```
"아까 code-refactor 에이전트가 하던 작업 이어서 해줘"
"그 에이전트한테 추가로 OrderService도 분석시켜줘"
```

## 위임 시 효과

에이전트에 위임하면 다음과 같은 이점이 있습니다.

### 1. 컨텍스트 보존

```
┌────────────────────────────────┐
│  메인 대화 컨텍스트             │
│                                │
│  사용자 대화 ← 결과 요약만 반환 │
│                                │
│  ┌──────────────────────────┐  │
│  │  에이전트 독립 컨텍스트    │  │
│  │                          │  │
│  │  파일 Read (수십 개)      │  │
│  │  Grep 결과 (수백 줄)      │  │
│  │  테스트 실행 로그 (대량)   │  │
│  │  재시도 과정 (반복)        │  │
│  │                          │  │
│  │  → 이 모든 출력이 메인을  │  │
│  │    소비하지 않음          │  │
│  └──────────────────────────┘  │
└────────────────────────────────┘
```

- 에이전트의 대량 작업 출력(테스트 로그, diff, Grep 결과 등)이 메인 대화를 소비하지 않음
- 메인 대화는 결과 요약만 받으므로 컨텍스트 윈도우를 효율적으로 사용
- 메인 대화에서 길고 복잡한 작업을 반복해도 컨텍스트가 넘치지 않음

### 2. 비용 최적화

| 에이전트 | 모델 | 이유 |
|----------|------|------|
| test-generator | Sonnet | 코드 분석 + 생성에 높은 능력 필요 |
| code-refactor | Sonnet | 코드 품질 판단에 높은 능력 필요 |
| debugger | Sonnet | 근본 원인 추론에 높은 능력 필요 |
| jira-updater | **Haiku** | 단순 API 호출, 빠르고 저렴 |

- 작업 복잡도에 맞는 모델을 선택하여 비용 절감
- Haiku로 충분한 작업은 빠르고 저렴하게 처리

### 3. 도구 제한 (안전성)

각 에이전트는 필요한 도구만 사용할 수 있습니다:

| 에이전트 | 허용 도구 | 제한 이유 |
|----------|----------|----------|
| jira-updater | Read, Glob, Grep, Bash | 코드 수정 불필요 (Write/Edit 차단) |

- 에이전트가 의도치 않은 작업을 수행하는 것을 방지
- 최소 권한 원칙 (Principle of Least Privilege) 적용

### 4. 병렬 작업

독립적인 에이전트를 동시에 실행할 수 있습니다:

```
"test-generator로 테스트 생성하면서 동시에 code-refactor로 코드 분석도 해줘"
```

- 각 에이전트가 독립 컨텍스트에서 동시 실행
- 서로 영향을 주지 않음

### 5. 전문화된 시스템 프롬프트

각 에이전트는 해당 작업에 최적화된 시스템 프롬프트를 가집니다:

- `test-generator`: given/when/then 패턴, Mockito 격리, 재시도 전략 등 테스트 전문 지식
- `debugger`: 에러 타입별 분석 전략, git blame 추적, 호출 체인 분석 등 디버깅 전문 지식
- `code-refactor`: CLAUDE.md 규칙 대조 체크리스트, 수정 패턴 매핑 등 리팩토링 전문 지식

메인 Claude가 모든 것을 알 필요 없이, 전문 에이전트가 해당 도메인에 집중합니다.

## 에이전트 상세  

### 1. test-generator - 테스트 자동 생성

**모델**: Sonnet | **도구**: Read, Write, Edit, Glob, Grep, Bash

대상 클래스를 분석하여 단위/통합 테스트를 자동 생성하고 실행합니다.

#### 동작 흐름

```
대상 파악
  │
  ├── 프로젝트 분석 (병렬)
  │   ├── 빌드 도구 감지 (Gradle / Maven)
  │   ├── 테스트 프레임워크 감지 (JUnit5, Mockito, AssertJ)
  │   ├── 기존 테스트 스타일 파악
  │   └── 대상 클래스 분석 (공개 메서드, 의존성)
  │
  ├── 테스트 코드 생성
  │   ├── 단위 테스트 (Mockito, given/when/then)
  │   └── 통합 테스트 (@SpringBootTest, @Transactional)
  │
  ├── 실행 + 자동 수정 (최대 3회 재시도)
  │
  └── git add + 결과 보고
```

#### 테스트 규칙

| 항목 | 규칙 |
|------|------|
| 메서드명 | `메서드명_조건_기대결과` |
| 패턴 | given/when/then |
| 단위 테스트 | `@SpringBootTest` 사용 금지, Mockito 격리 |
| 통합 테스트 | `@SpringBootTest` + `@Transactional` 롤백 |

---

### 2. code-refactor - 코드 리팩토링

**모델**: Sonnet | **도구**: Read, Write, Edit, Glob, Grep, Bash

CLAUDE.md 규칙 기반으로 코드 스멜을 감지하고 리팩토링합니다.

#### 감지 항목

| 심각도 | 항목 | 수정 방법 |
|--------|------|----------|
| 🔴 높음 | `@Data` | `@Getter` + `@NoArgsConstructor` + `@Builder` |
| 🔴 높음 | `System.out.println` | SLF4J 로거 |
| 🔴 높음 | `FetchType.EAGER` | `FetchType.LAZY` |
| 🔴 높음 | Entity API 노출 | ResponseDTO 분리 |
| 🔴 높음 | 중첩 루프 O(N²) | Map 검색 / Stream |
| ⚠️ 중간 | 매직 넘버 | `static final` 상수 |
| ⚠️ 중간 | if-else 3개+ 연쇄 | switch / Enum / Map |
| ⚠️ 중간 | 파라미터 3개 초과 | 객체 그룹화 |

#### 동작 흐름

```
대상 분석 → 코드 스멜 보고서 → 사용자 승인
  │
  ├── "전체 적용" → 순차 수정 + 테스트 검증
  ├── "항목 선택" → 선택 항목만 수정
  └── "취소" → 종료
```

수정 후 기존 테스트를 실행하여 검증하며, 테스트가 깨지면 즉시 롤백합니다.

---

### 3. debugger - 에러 디버깅

**모델**: Sonnet | **도구**: Read, Write, Edit, Glob, Grep, Bash

스택 트레이스를 분석하여 근본 원인을 추적하고 수정합니다.

#### 에러 타입별 분석

| 에러 | 분석 방법 |
|------|----------|
| NullPointerException | null 체크 누락, Optional 미사용 |
| LazyInitializationException | 트랜잭션 범위, Fetch 전략 |
| SQL (DataIntegrity) | 쿼리 분석, 제약 조건 |
| Deadlock | 락, 트랜잭션 격리 수준 |
| ClassCastException | 타입 변환 체인 |
| StackOverflowError | 순환 참조, 재귀 호출 |

#### 동작 흐름

```
에러 정보 수집
  │
  ├── 코드 수집 (병렬)
  │   ├── 스택 트레이스 파싱 (파일:라인)
  │   ├── 소스 코드 Read
  │   ├── git log / git blame
  │   └── 호출 체인 Grep
  │
  ├── 원인 분석 → 수정 제안 보고서
  │
  ├── 사용자 승인: "수정 적용 / 분석만 / 취소"
  │
  └── 수정 + 테스트 검증
```

---

### 4. jira-updater - Jira 이슈 관리

**모델**: Haiku | **도구**: Read, Glob, Grep, Bash | **MCP**: atlassian

현재 브랜치에서 이슈 키를 감지하여 상태를 전환합니다.

#### 동작 흐름

```
브랜치명 → 이슈 키 추출 (예: feature/TECH-1234-auth → TECH-1234)
  │
  ├── MCP: 이슈 조회 + 전환 가능 상태
  ├── 사용자 선택: 전환할 상태
  ├── MCP: 상태 전환
  │
  ├── 코멘트: "자동 생성 / 직접 입력 / 건너뛰기"
  └── 완료 보고
```

MCP 실패 시 curl REST API로 자동 폴백합니다.

## 공통 패턴

### 재시도 전략

모든 에이전트는 실패 시 동일한 재시도 패턴을 사용합니다:

```
1차 → 실패 → 에러 분석 → 수정 → 재시도
2차 → 실패 → 다른 접근 → 재시도
3차 → 실패 → 사용자에게 질문
              ├── "스킵 후 계속"
              ├── "방향 직접 지정"
              └── "작업 중단"
```

### Git 규칙

- 모든 에이전트는 `git add`까지만 수행합니다
- **commit은 절대 수행하지 않습니다** (사용자 요청 시에만)

### 컨텍스트 격리

에이전트는 독립 컨텍스트에서 실행되므로:
- 메인 대화의 컨텍스트를 소비하지 않음
- 대량 출력 (테스트 결과, diff 등)이 메인 컨텍스트에 영향을 주지 않음
- 결과 요약만 메인 대화로 반환됨

## 활용 예시

### 기능 구현 워크플로우

```
1. 기능 구현
2. "테스트 생성해줘"              → test-generator 자동 위임
3. "리팩토링 필요한지 확인해줘"    → code-refactor 자동 위임
4. /self-review                   → 셀프 리뷰 (Command)
5. "Jira 상태 업데이트해줘"       → jira-updater 자동 위임
6. /pr                            → PR 생성 (Command)
```

### 버그 수정 워크플로우

```
1. "이 에러 분석해줘: {스택 트레이스}" → debugger 자동 위임
2. "수정 검증 테스트 만들어줘"        → test-generator 자동 위임
3. "Jira 이슈 업데이트"             → jira-updater 자동 위임
```
