# 실무 에이전트 가이드

Claude Code에서 사용할 수 있는 **Sub-Agent**와 **Agent Team** 가이드입니다.
각 에이전트는 독립 컨텍스트에서 전용 시스템 프롬프트와 도구 접근 권한을 가지고 자율적으로 작업을 수행합니다.

## Sub-Agent vs Command 차이점

```
┌─────────────────────────────────────────────────────┐
│  Command (슬래시 명령어)                              │
│                                                       │
│  - 메인 대화 컨텍스트에서 실행                         │
│  - /pr, /self-review 등                              │
│  - 사용자가 직접 호출                                  │
│  - 메인 컨텍스트를 소비함                              │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  Sub-Agent (에이전트)                                 │
│                                                       │
│  - 독립 컨텍스트 윈도우에서 실행                       │
│  - Claude가 작업 유형에 따라 자동 위임                 │
│  - 명시적 요청도 가능                                  │
│  - 메인 컨텍스트를 보존함 (결과 요약만 반환)            │
│  - 모델, 도구, 권한을 독립적으로 설정                  │
│  - 백그라운드 실행 가능                                │
└─────────────────────────────────────────────────────┘
```

## Sub-Agent vs Agent Team 차이점

Sub-Agent와 Agent Team은 근본적으로 다른 협업 모델입니다.

### 동작 원리

```
┌─────────────────────────────────────────────────────────────────┐
│  Sub-Agent (서브 에이전트)                                       │
│                                                                   │
│  메인 Claude ──→ Task 도구로 위임 ──→ 독립 컨텍스트에서 작업      │
│       ↑                                        │                  │
│       └──────── 결과 요약만 단방향 반환 ←────────┘                  │
│                                                                   │
│  - 메인이 "감독자", Sub-Agent는 "작업자"                           │
│  - 메인 컨텍스트 안에서 Task 도구로 생성                            │
│  - Sub-Agent 간 서로 통신 불가 (각각 메인에만 보고)                  │
│  - 결과를 메인이 받아서 다음 작업 결정                              │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Agent Team (팀 에이전트)                                        │
│                                                                   │
│  Team Lead ──→ 팀 멤버 N명 생성 ──→ 각각 독립 인스턴스             │
│       ↑              ↕ 양방향 소통        ↕                       │
│       └──── 공유 작업 목록에서 자체 조율 ────┘                      │
│                                                                   │
│  - Lead가 "PM", Teammate가 "팀원"                                 │
│  - 각 멤버가 완전 독립 Claude 인스턴스                              │
│  - 멤버 간 직접 메시지 전달 가능 (Shift+Down)                       │
│  - 공유 작업 목록에서 자기가 할 일을 스스로 선택                     │
└─────────────────────────────────────────────────────────────────┘
```

### 핵심 비교

| 특성 | Sub-Agent | Agent Team |
|------|-----------|------------|
| **컨텍스트** | 메인 내부에서 독립 프로세스 생성 | 각 멤버가 완전 독립 인스턴스 |
| **통신 방향** | 단방향 (결과만 메인으로) | 양방향 (멤버 간 직접 소통) |
| **조정 방식** | 메인이 모든 작업 관리/결정 | 공유 작업 목록에서 자체 조정 |
| **파일 격리** | 같은 워킹 디렉토리 공유 | Worktree로 독립 디렉토리 가능 |
| **실행 모드** | foreground / background | In-Process / Split-Pane (tmux) |
| **활성화** | 기본 내장 | 실험적 기능 (환경변수 필요) |
| **적합한 작업** | 자체 포함된 단일 작업 | 복잡한 병렬 탐구/개발 |
| **비용** | 낮음 (~$1.0/작업 x3개) | 높음 (~$3.5/작업 x3명) |

### 선택 기준

```
결과만 필요하고, 단일 작업인가?
  → Yes: Sub-Agent
  → No: 팀 멤버 간 발견사항 공유가 필요한가?
         → Yes: Agent Team
         → No: Sub-Agent 여러 개 병렬 실행
```

| 상황 | 추천 | 이유 |
|------|------|------|
| 테스트 생성 | Sub-Agent | 단일 목적, 결과만 필요 |
| 코드 리팩토링 | Sub-Agent | 독립 작업, 순차 적용 |
| 에러 디버깅 | Sub-Agent | 단일 추적 작업 |
| 다관점 코드 리뷰 | **Agent Team** | 관점 간 크로스체크 필요 |
| 모듈별 동시 개발 | **Agent Team** | 파일 충돌 격리, Worktree 필요 |
| 대규모 마이그레이션 | **Agent Team** | Fan-out 병렬 처리 |

### 비용 비교

```
단독 세션:         ~$0.50/작업
Sub-Agent 3개:     ~$1.00/작업 (x2)
Agent Team 3명:    ~$3.50/작업 (x7, Plan 모드 포함)
```

Agent Team은 각 멤버가 완전 독립 인스턴스이므로 토큰 소비가 크고, 팀 조율에도 추가 토큰이 소모됩니다. 5명 초과 시 조율 오버헤드(N²)가 병렬화 이점을 상회합니다.

## 위임 방식

에이전트에게 작업을 위임하는 방법은 **자동 위임**과 **수동 위임** 두 가지가 있습니다.

### 자동 위임

Claude가 사용자 요청의 의도와 에이전트의 `description`을 비교하여 자동으로 적절한 에이전트에 위임합니다.

```
사용자: "UserService의 테스트를 만들어줘"
→ Claude가 test-generator 에이전트의 description과 매칭
→ 자동으로 에이전트에 위임
→ 에이전트가 독립 컨텍스트에서 작업 수행
→ 결과 요약만 메인 대화로 반환
```

### 수동 위임

에이전트 이름을 직접 지정하여 명시적으로 위임할 수 있습니다.
자동 위임이 되지 않거나, 특정 에이전트를 지정하고 싶을 때 사용합니다.

```
"test-generator 에이전트로 UserService 테스트를 생성해줘"
"debugger 에이전트를 사용해서 이 에러를 분석해줘"
"code-refactor 에이전트한테 변경 파일 리팩토링 맡겨줘"
```

### 실행 모드

#### 포그라운드 (기본)

메인 대화를 블로킹하고 에이전트가 작업을 완료할 때까지 대기합니다.
사용자에게 권한 확인이나 질문을 전달할 수 있습니다.

#### 백그라운드

에이전트가 백그라운드에서 작업하는 동안 메인 대화를 계속할 수 있습니다.

```
"테스트 생성을 백그라운드에서 진행해줘"
```

- 실행 전 필요한 도구 권한을 미리 승인받음
- 실행 중 Ctrl+B로 포그라운드 ↔ 백그라운드 전환 가능
- 작업 완료 시 결과를 메인 대화로 반환

#### 이어서 작업 (Resume)

에이전트가 완료한 이후에도 이전 컨텍스트를 유지한 채 작업을 이어갈 수 있습니다.

```
"아까 code-refactor 에이전트가 하던 작업 이어서 해줘"
"그 에이전트한테 추가로 OrderService도 분석시켜줘"
```

## 위임 시 효과

에이전트에 위임하면 다음과 같은 이점이 있습니다.

### 1. 컨텍스트 보존

```
┌────────────────────────────────┐
│  메인 대화 컨텍스트             │
│                                │
│  사용자 대화 ← 결과 요약만 반환 │
│                                │
│  ┌──────────────────────────┐  │
│  │  에이전트 독립 컨텍스트    │  │
│  │                          │  │
│  │  파일 Read (수십 개)      │  │
│  │  Grep 결과 (수백 줄)      │  │
│  │  테스트 실행 로그 (대량)   │  │
│  │  재시도 과정 (반복)        │  │
│  │                          │  │
│  │  → 이 모든 출력이 메인을  │  │
│  │    소비하지 않음          │  │
│  └──────────────────────────┘  │
└────────────────────────────────┘
```

- 에이전트의 대량 작업 출력(테스트 로그, diff, Grep 결과 등)이 메인 대화를 소비하지 않음
- 메인 대화는 결과 요약만 받으므로 컨텍스트 윈도우를 효율적으로 사용
- 메인 대화에서 길고 복잡한 작업을 반복해도 컨텍스트가 넘치지 않음

### 2. 비용 최적화

| 에이전트 | 모델 | 이유 |
|----------|------|------|
| test-generator | Sonnet | 코드 분석 + 생성에 높은 능력 필요 |
| code-refactor | Sonnet | 코드 품질 판단에 높은 능력 필요 |
| debugger | Sonnet | 근본 원인 추론에 높은 능력 필요 |
| jira-updater | **Haiku** | 단순 API 호출, 빠르고 저렴 |

- 작업 복잡도에 맞는 모델을 선택하여 비용 절감
- Haiku로 충분한 작업은 빠르고 저렴하게 처리

### 3. 도구 제한 (안전성)

각 에이전트는 필요한 도구만 사용할 수 있습니다:

| 에이전트 | 허용 도구 | 제한 이유 |
|----------|----------|----------|
| jira-updater | Read, Glob, Grep, Bash | 코드 수정 불필요 (Write/Edit 차단) |

- 에이전트가 의도치 않은 작업을 수행하는 것을 방지
- 최소 권한 원칙 (Principle of Least Privilege) 적용

### 4. 병렬 작업

독립적인 에이전트를 동시에 실행할 수 있습니다:

```
"test-generator로 테스트 생성하면서 동시에 code-refactor로 코드 분석도 해줘"
```

- 각 에이전트가 독립 컨텍스트에서 동시 실행
- 서로 영향을 주지 않음

### 5. 전문화된 시스템 프롬프트

각 에이전트는 해당 작업에 최적화된 시스템 프롬프트를 가집니다:

- `test-generator`: given/when/then 패턴, Mockito 격리, 재시도 전략 등 테스트 전문 지식
- `debugger`: 에러 타입별 분석 전략, git blame 추적, 호출 체인 분석 등 디버깅 전문 지식
- `code-refactor`: CLAUDE.md 규칙 대조 체크리스트, 수정 패턴 매핑 등 리팩토링 전문 지식

메인 Claude가 모든 것을 알 필요 없이, 전문 에이전트가 해당 도메인에 집중합니다.

## 에이전트 상세  

### 1. test-generator - 테스트 자동 생성

**모델**: Sonnet | **도구**: Read, Write, Edit, Glob, Grep, Bash

대상 클래스를 분석하여 단위/통합 테스트를 자동 생성하고 실행합니다.

#### 동작 흐름

```
대상 파악
  │
  ├── 프로젝트 분석 (병렬)
  │   ├── 빌드 도구 감지 (Gradle / Maven)
  │   ├── 테스트 프레임워크 감지 (JUnit5, Mockito, AssertJ)
  │   ├── 기존 테스트 스타일 파악
  │   └── 대상 클래스 분석 (공개 메서드, 의존성)
  │
  ├── 테스트 코드 생성
  │   ├── 단위 테스트 (Mockito, given/when/then)
  │   └── 통합 테스트 (@SpringBootTest, @Transactional)
  │
  ├── 실행 + 자동 수정 (최대 3회 재시도)
  │
  └── git add + 결과 보고
```

#### 테스트 규칙

| 항목 | 규칙 |
|------|------|
| 메서드명 | `메서드명_조건_기대결과` |
| 패턴 | given/when/then |
| 단위 테스트 | `@SpringBootTest` 사용 금지, Mockito 격리 |
| 통합 테스트 | `@SpringBootTest` + `@Transactional` 롤백 |

---

### 2. code-refactor - 코드 리팩토링

**모델**: Sonnet | **도구**: Read, Write, Edit, Glob, Grep, Bash

CLAUDE.md 규칙 기반으로 코드 스멜을 감지하고 리팩토링합니다.

#### 감지 항목

| 심각도 | 항목 | 수정 방법 |
|--------|------|----------|
| 🔴 높음 | `@Data` | `@Getter` + `@NoArgsConstructor` + `@Builder` |
| 🔴 높음 | `System.out.println` | SLF4J 로거 |
| 🔴 높음 | `FetchType.EAGER` | `FetchType.LAZY` |
| 🔴 높음 | Entity API 노출 | ResponseDTO 분리 |
| 🔴 높음 | 중첩 루프 O(N²) | Map 검색 / Stream |
| ⚠️ 중간 | 매직 넘버 | `static final` 상수 |
| ⚠️ 중간 | if-else 3개+ 연쇄 | switch / Enum / Map |
| ⚠️ 중간 | 파라미터 3개 초과 | 객체 그룹화 |

#### 동작 흐름

```
대상 분석 → 코드 스멜 보고서 → 사용자 승인
  │
  ├── "전체 적용" → 순차 수정 + 테스트 검증
  ├── "항목 선택" → 선택 항목만 수정
  └── "취소" → 종료
```

수정 후 기존 테스트를 실행하여 검증하며, 테스트가 깨지면 즉시 롤백합니다.

---

### 3. debugger - 에러 디버깅

**모델**: Sonnet | **도구**: Read, Write, Edit, Glob, Grep, Bash

스택 트레이스를 분석하여 근본 원인을 추적하고 수정합니다.

#### 에러 타입별 분석

| 에러 | 분석 방법 |
|------|----------|
| NullPointerException | null 체크 누락, Optional 미사용 |
| LazyInitializationException | 트랜잭션 범위, Fetch 전략 |
| SQL (DataIntegrity) | 쿼리 분석, 제약 조건 |
| Deadlock | 락, 트랜잭션 격리 수준 |
| ClassCastException | 타입 변환 체인 |
| StackOverflowError | 순환 참조, 재귀 호출 |

#### 동작 흐름

```
에러 정보 수집
  │
  ├── 코드 수집 (병렬)
  │   ├── 스택 트레이스 파싱 (파일:라인)
  │   ├── 소스 코드 Read
  │   ├── git log / git blame
  │   └── 호출 체인 Grep
  │
  ├── 원인 분석 → 수정 제안 보고서
  │
  ├── 사용자 승인: "수정 적용 / 분석만 / 취소"
  │
  └── 수정 + 테스트 검증
```

---

### 4. jira-updater - Jira 이슈 관리

**모델**: Haiku | **도구**: Read, Glob, Grep, Bash | **MCP**: atlassian

현재 브랜치에서 이슈 키를 감지하여 상태를 전환합니다.

#### 동작 흐름

```
브랜치명 → 이슈 키 추출 (예: feature/TECH-1234-auth → TECH-1234)
  │
  ├── MCP: 이슈 조회 + 전환 가능 상태
  ├── 사용자 선택: 전환할 상태
  ├── MCP: 상태 전환
  │
  ├── 코멘트: "자동 생성 / 직접 입력 / 건너뛰기"
  └── 완료 보고
```

MCP 실패 시 curl REST API로 자동 폴백합니다.

## 공통 패턴

### 재시도 전략

모든 에이전트는 실패 시 동일한 재시도 패턴을 사용합니다:

```
1차 → 실패 → 에러 분석 → 수정 → 재시도
2차 → 실패 → 다른 접근 → 재시도
3차 → 실패 → 사용자에게 질문
              ├── "스킵 후 계속"
              ├── "방향 직접 지정"
              └── "작업 중단"
```

### Git 규칙

- 모든 에이전트는 `git add`까지만 수행합니다
- **commit은 절대 수행하지 않습니다** (사용자 요청 시에만)

### 컨텍스트 격리

에이전트는 독립 컨텍스트에서 실행되므로:
- 메인 대화의 컨텍스트를 소비하지 않음
- 대량 출력 (테스트 결과, diff 등)이 메인 컨텍스트에 영향을 주지 않음
- 결과 요약만 메인 대화로 반환됨

## 활용 예시

### 기능 구현 워크플로우

```
1. 기능 구현
2. "테스트 생성해줘"              → test-generator 자동 위임
3. "리팩토링 필요한지 확인해줘"    → code-refactor 자동 위임
4. /self-review                   → 셀프 리뷰 (Command)
5. "Jira 상태 업데이트해줘"       → jira-updater 자동 위임
6. /pr                            → PR 생성 (Command)
```

### 버그 수정 워크플로우

```
1. "이 에러 분석해줘: {스택 트레이스}" → debugger 자동 위임
2. "수정 검증 테스트 만들어줘"        → test-generator 자동 위임
3. "Jira 이슈 업데이트"             → jira-updater 자동 위임
```

## 팀 에이전트 워크플로우

### Agent Team 활성화

Agent Team은 실험적 기능으로, 명시적 활성화가 필요합니다.

```bash
# 환경 변수 방식
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# 또는 settings.json 방식
# ~/.claude/settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### `/work-plan-start` 팀 워크플로우

WORK-SPEC.md 기반의 복잡한 작업(파일 3개 이상 수정) 시 Sub-Agent를 팀처럼 조율합니다.

```
Phase 1: 탐색 + 설계 (병렬, foreground - 결과 대기)
├─ [Explore]  코드베이스 구조 파악, 영향 범위 분석
└─ [Plan]     구현 전략 설계, 파일별 변경 계획

Phase 2: 구현 + 테스트 (병렬)
├─ [Main]            핵심 코드 수정 (Phase 1 결과 기반)
└─ [test-generator]  테스트 자동 생성 (background)

Phase 3: 검증 + 문서화 (병렬)
├─ [code-refactor]   코드 품질 리뷰 (background)
└─ [Main]            ARCHITECTURE.md, SPEC.md 작성
```

| Phase | 에이전트 | 입력 | 실행 모드 |
|-------|----------|------|-----------|
| 1 | Explore | WORK-SPEC.md의 변경 대상 파일 + 키워드 | foreground |
| 1 | Plan | WORK-SPEC.md 전체 내용 | foreground |
| 2 | Main | Phase 1 결과 | foreground |
| 2 | test-generator | Plan 결과 + 변경 파일 경로 | background |
| 3 | code-refactor | Phase 2 변경 파일 목록 | background |
| 3 | Main | Phase 2 완료 코드 기반 | foreground |

**적용 조건**: WORK-SPEC.md가 존재하고 파일 3개 이상 수정 시 자동 적용. 단순 작업은 Main 단독 처리.

### `/team-review` 팀 워크플로우

4명의 전문 리뷰어를 병렬로 실행하여 다관점 코드 리뷰를 수행합니다.

```
Team Lead (Main)
  │
  ├── review-performance   (성능: N+1 쿼리, 고비용 객체, O(N²) 루프)
  ├── review-security      (보안: OWASP Top 10, SQL Injection, 시크릿)
  ├── review-test-coverage (테스트: 커버리지, 누락 시나리오, 품질)
  └── review-convention    (컨벤션: CLAUDE.md 규칙, SOLID, 네이밍)

  → 4개 Task를 하나의 메시지에서 동시 호출
  → 각 리뷰어는 읽기 전용 도구만 사용 (Write/Edit 차단)
  → 결과를 TEAM-REVIEW.md로 통합
```

| 상황 | 추천 커맨드 | 이유 |
|------|------------|------|
| 소규모 변경 (파일 1~3개) | `/self-review` | 오버헤드 최소 |
| 중요 기능/PR 전 | `/team-review` | 다관점 검증 |
| 다른 사람의 PR 리뷰 | `/review-pr` | PR 번호 기반 |

### Sub-Agent 병렬 실행 vs Agent Team

이 프로젝트에서 `/work-plan-start`와 `/team-review`는 **Sub-Agent 병렬 실행** 방식을 사용합니다. 엄밀한 Agent Team(환경변수 활성화 필요)과는 다릅니다.

```
Sub-Agent 병렬 실행 (현재 사용 중)
─────────────────────────────────
Main이 Task 도구로 여러 Sub-Agent를 동시 호출
  → 각각 독립 컨텍스트에서 작업
  → 결과를 Main이 수집하여 통합
  → 멤버 간 직접 소통 없음
  → 추가 환경 설정 불필요

Agent Team (실험적 기능)
─────────────────────────
환경변수 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 필요
  → 각 멤버가 완전 독립 인스턴스
  → 공유 작업 목록에서 자체 조율
  → 멤버 간 양방향 소통 가능
  → tmux Split-Pane으로 동시 모니터링
```

현재 워크플로우는 Sub-Agent 병렬 실행으로도 충분하며, 멤버 간 발견사항 공유가 필요한 경우에만 Agent Team을 고려합니다.

## 파일 위치

### 프로젝트 에이전트 (이 저장소 전용, 우선순위 2)

```
.claude/agents/
├── test-generator.md
├── code-refactor.md
├── debugger.md
└── jira-updater.md
```

### 글로벌 에이전트 (모든 프로젝트 공용, 우선순위 3)

```
~/.claude/agents/
├── test-generator.md
├── code-refactor.md
├── debugger.md
└── jira-updater.md
```

동일 이름의 에이전트가 두 곳에 있으면 프로젝트 에이전트가 우선합니다.
프로젝트별 커스터마이징이 필요하면 프로젝트 에이전트를 수정하세요.
