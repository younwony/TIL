# Part 8: 테스트 전략 (Testing Strategies)

## 핵심 내용

### QA는 오류를 찾지 못해야 한다

- 개발 단계에서 모든 오류를 잡아야 한다
- QA에게 버그를 넘기는 것은 프로답지 못하다

### QA는 같은 팀

QA의 역할:
- **명세 서술**: 요구사항을 테스트로 표현
- **특징 묘사**: 시스템의 실제 동작을 기록

### 테스트 자동화 피라미드

```
        /\
       /  \
      / 수동 \        ← 탐색적 테스트
     /--------\
    /  시스템  \      ← 시스템 테스트 (10%)
   /------------\
  /    통합      \    ← 통합 테스트
 /----------------\
/    컴포넌트      \   ← 컴포넌트 테스트 (50%)
/------------------\
      단위 테스트       ← 단위 테스트 (90%+)
```

### 1. 단위 테스트 (Unit Tests)

- **커버리지**: 90~99%
- **목적**: 개별 코드 유닛의 정확성 검증
- **작성자**: 개발자
- **실행 주기**: 매우 빈번 (초 단위)

```java
@Test
void testAddition() {
    Calculator calc = new Calculator();
    assertEquals(4, calc.add(2, 2));
}
```

### 2. 컴포넌트 테스트 (Component Tests)

- **커버리지**: 시스템의 약 절반
- **목적**: 개별 컴포넌트의 동작 검증
- **작성자**: QA와 비즈니스, 개발자 지원
- **도구**: FitNesse, JBehave, Cucumber

```gherkin
Given 사용자가 로그인했을 때
When 상품을 장바구니에 추가하면
Then 장바구니에 상품이 표시된다
```

### 3. 통합 테스트 (Integration Tests)

- **목적**: 컴포넌트 간의 상호작용 검증
- **작성자**: 시스템 아키텍트, 리드 개발자
- **실행 주기**: 주기적 (하루에 여러 번)

### 4. 시스템 테스트 (System Tests)

- **커버리지**: 시스템의 약 10%
- **목적**: 전체 시스템의 통합 동작 검증
- **작성자**: 시스템 아키텍트, 기술 리드
- **실행 주기**: 비교적 드묾 (하루에 한 번 등)

### 5. 수동 탐색 테스트 (Manual Exploratory Tests)

- **목적**: 자동화된 테스트로 잡지 못하는 문제 발견
- **특징**: 인간의 창의력 활용
- **방법**: 시나리오 없이 자유롭게 탐색

## 결론

- 모든 계층의 테스트가 필요하다
- 테스트 피라미드를 유지한다
- 자동화와 수동 테스트의 균형을 맞춘다
