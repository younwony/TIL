# Part 7: 인수 테스트 (Acceptance Testing)

## 핵심 내용

### 요구사항 관련 의사소통

- 요구사항은 불완전하고 모호하다
- 의사소통이 핵심이다

### 시기상조의 정밀도

- 초기에 너무 상세한 요구사항을 요구하지 마라
- 정밀도는 시간이 지나면서 높아진다

### 불확실성의 원칙

- 요구사항을 관찰하는 순간 요구사항이 변한다 (관찰자 효과)
- 고객이 실제 동작을 보면 원하는 것이 바뀐다

### 불안한 추정

- 추정에는 오차범위가 필요하다
- 정확한 추정은 불가능하다

### 때늦은 모호함

- 정밀도는 가능한 늦게 높인다
- 너무 일찍 결정하면 변경 비용이 크다

### 인수 테스트 정의

- **인수 테스트**: 요구사항이 충족되었는지 확인하는 테스트
- 비즈니스와 개발자가 함께 작성한다
- "완료"의 정의가 된다

### 완료에 대한 정의

```
완료 = 모든 자동화된 인수 테스트 통과
```

### 의사소통

- 인수 테스트는 개발자와 비즈니스 간의 의사소통 도구다
- 오해를 줄이고 명확성을 높인다

### 자동화

인수 테스트 도구:
- FitNesse
- Cucumber
- Robot Framework
- Selenium

### 누가, 언제 인수 테스트를 작성하는가?

- **누가**: QA, BA, 개발자가 협력하여 작성
- **언제**: 구현 전에 작성 (이터레이션 초반)

### 개발자의 역할

- 인수 테스트를 자동화한다
- 테스트가 통과하도록 코드를 작성한다
- 테스트 결과를 모니터링한다

### 테스트 협상과 수동적 공격성

- 불합리한 테스트에는 협상한다
- 수동적으로 순응하지 않는다

### 인수 테스트와 단위 테스트의 차이

| 구분 | 인수 테스트 | 단위 테스트 |
|------|-----------|-----------|
| 목적 | 비즈니스 요구사항 검증 | 기술적 정확성 검증 |
| 작성자 | QA, BA, 개발자 | 개발자 |
| 범위 | 전체 시스템 | 개별 컴포넌트 |
| 관점 | 사용자 관점 | 개발자 관점 |

### GUI 및 다른 문제점

- GUI 테스트는 불안정하다
- 올바른 인터페이스를 통해 테스트한다

### 지속적 통합 (CI)

- 인수 테스트는 CI 파이프라인에 포함한다
- 테스트 실패 시 빌드를 중단한다

### 출시를 멈춰라

- 테스트가 실패하면 출시하지 않는다
- 깨진 빌드를 방치하지 않는다
