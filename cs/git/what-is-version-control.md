# 버전 관리란 (What is Version Control)

## 왜 알아야 하는가?

버전 관리는 모든 개발 프로젝트의 기반입니다. 코드 변경 이력 추적, 협업, 롤백 등 필수 기능을 제공합니다. Git은 현대 개발의 표준이며, 면접에서도 Git 사용 경험과 이해도를 자주 질문합니다.

## 핵심 개념

- **저장소 (Repository)**: 버전 관리되는 파일들의 저장 공간
- **커밋 (Commit)**: 변경 사항의 스냅샷
- **브랜치 (Branch)**: 독립적인 개발 라인
- **머지 (Merge)**: 브랜치 통합

## 쉽게 이해하기

**버전 관리**를 문서 작업에 비유할 수 있습니다.

버전 관리 없이:
```
보고서_최종.docx
보고서_최종_수정.docx
보고서_진짜최종.docx
보고서_진짜진짜최종_v2.docx
```

버전 관리와 함께:
```
보고서.docx
├── v1.0: 초안 작성 (2024-01-01)
├── v1.1: 서론 수정 (2024-01-02)
├── v2.0: 결론 추가 (2024-01-03)
└── v2.1: 오타 수정 (2024-01-04)
```

## 상세 설명

### 버전 관리가 필요한 이유

| 문제 | 해결 |
|------|------|
| "누가 이 코드를 수정했지?" | 이력 추적 |
| "어제 잘 되던 게 왜 안 되지?" | 롤백 |
| "동시에 같은 파일 수정 중" | 머지 |
| "이 기능 따로 개발하고 싶어" | 브랜치 |
| "서버 날아가면 어떡하지?" | 분산 백업 |

### 중앙집중식 vs 분산

```
중앙집중식 (CVCS - SVN):
    ┌───────────────────┐
    │  Central Server   │
    │   (저장소)         │
    └────────┬──────────┘
             │
    ┌────────┼────────┐
    │        │        │
┌───▼───┐ ┌──▼───┐ ┌──▼───┐
│ 개발자A│ │개발자B│ │개발자C│
│(작업본)│ │(작업본)│ │(작업본)│
└───────┘ └──────┘ └──────┘

분산 (DVCS - Git):
┌───────────────────┐
│  Remote Server    │
│   (원격 저장소)    │
└────────┬──────────┘
         │ push/pull
┌────────┼────────┐
│        │        │
┌───▼───┐ ┌──▼───┐ ┌──▼───┐
│ 개발자A│ │개발자B│ │개발자C│
│(전체복제)│ │(전체복제)│ │(전체복제)│
└───────┘ └──────┘ └──────┘
각자 완전한 저장소 보유
```

| 구분 | CVCS (SVN) | DVCS (Git) |
|------|-----------|-----------|
| 저장소 | 중앙 1개 | 각자 전체 복제 |
| 오프라인 작업 | 불가 | 가능 |
| 속도 | 느림 (네트워크) | 빠름 (로컬) |
| 백업 | 중앙 서버 의존 | 모든 복제본이 백업 |

### Git의 3가지 영역

```
Working Directory     Staging Area        Repository
  (작업 디렉토리)         (스테이징)           (저장소)
       │                    │                   │
       │     git add        │    git commit     │
       │ ─────────────────► │ ────────────────► │
       │                    │                   │
       │                    │                   │
       │ ◄─────────────────────────────────── │
       │              git checkout              │
```

### 기본 워크플로우

```bash
# 1. 저장소 복제
git clone https://github.com/user/repo.git

# 2. 브랜치 생성
git checkout -b feature/new-feature

# 3. 파일 수정 후 스테이징
git add file.txt

# 4. 커밋
git commit -m "Add new feature"

# 5. 원격 저장소에 푸시
git push origin feature/new-feature

# 6. Pull Request 생성 및 머지
```

### 브랜치 개념

```
      main
        │
        ●──────●──────●──────●──────●
                \            /
                 ●──●──●──●
                feature-branch

브랜치: 독립적인 개발 라인
머지: 브랜치 통합
```

### Git vs 다른 VCS

| VCS | 방식 | 특징 |
|-----|------|------|
| Git | 분산 | 빠름, 브랜치 강력, 가장 인기 |
| SVN | 중앙집중 | 간단, 큰 바이너리 처리 |
| Mercurial | 분산 | Git과 유사, 더 단순 |
| Perforce | 중앙집중 | 대용량 파일, 게임 업계 |

## 트레이드오프

| 장점 | 단점 |
|------|------|
| 변경 이력 추적 | 학습 곡선 |
| 협업 용이 | 충돌 해결 필요 |
| 롤백 가능 | 저장소 크기 증가 |
| 브랜치로 격리 작업 | 복잡한 이력 관리 |

## 면접 예상 질문

### Q: Git을 사용하는 이유는?

A: (1) **분산 저장소**: 로컬에 전체 이력이 있어 오프라인 작업, 빠른 속도 (2) **강력한 브랜치**: 브랜치 생성/전환이 가볍고 빠름 (3) **무결성**: SHA-1 해시로 모든 변경 검증 (4) **협업**: Pull Request, Code Review 지원. **핵심**: 분산 구조와 브랜치 덕분에 효율적인 협업이 가능합니다.

### Q: Commit과 Push의 차이는?

A: **Commit**: 로컬 저장소에 변경 사항 저장 (스냅샷). **Push**: 로컬 커밋을 원격 저장소에 업로드. **분리 이유**: 오프라인에서도 커밋 가능, 여러 커밋을 모아서 푸시 가능, 커밋 정리 후 공유 가능.
