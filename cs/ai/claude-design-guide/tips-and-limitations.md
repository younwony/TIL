# Claude Design 활용 팁과 한계

> 이 문서는 [메인 가이드](./README.md)의 보조 문서로, Claude Design을 실전에서 사용할 때의 효과적인 프롬프트 패턴, 4가지 수정 수단의 적절한 사용 시점, 알려진 한계와 우회 방법을 정리합니다.

---

## 1. 효과적인 프롬프트 4요소

Anthropic 공식 가이드가 권장하는 프롬프트 구성 요소다.

| 요소 | 의미 | 좋은 예 | 나쁜 예 |
|------|------|--------|--------|
| **목표** | 무엇을 만드는가 | "월별 매출 대시보드를 만들고 싶다" | "예쁜 거 만들어줘" |
| **레이아웃** | 어떻게 배치하는가 | "상단에 KPI 카드 4개, 하단에 라인 차트" | "보기 좋게" |
| **콘텐츠** | 어떤 정보가 들어가는가 | "필터: 기간 / 제품군 / 지역" | "필요한 거" |
| **대상** | 누가 사용하는가 | "운영팀이 매주 모니터링" | (생략) |

### 통합 예시

> "운영팀이 매주 모니터링할 월별 매출 대시보드를 만들어줘. 상단에 KPI 카드 4개(총매출/전월대비/이익률/이상치 건수), 하단에 라인 차트와 지역별 막대 차트. 필터는 기간/제품군/지역. 회사 디자인 시스템을 사용하고 다크 모드 우선."

## 2. 점진적 개선 패턴

### 권장 순서: 단순 → 복잡

영상 발표자도 슬라이드덱을 한 번에 완성하지 않고 **2단계 프롬프트**로 만들었다:

1. **1차**: 콘텐츠 + 디자인 시스템만 — "이 스크립트로 PPT 슬라이드를 만들어 달라"
2. **2차**: 시각 효과 추가 — "전체적인 내용은 유지하되 WebGL 쉐이더 + 추상 도형 + JSX 애니메이션으로 개선해 달라"

> 한 번에 모든 요구를 넣으면 결과 품질이 평균화된다. **콘셉트 확정 → 비주얼 개선** 순서가 안정적이다.

### Save and Try Different Approach

공식 도움말의 **버전 관리** 기능을 활용:

```
"Save what we have and try a completely different approach"
```

- 현재 결과를 저장하고 완전히 다른 방향을 탐색
- 마음에 안 들면 이전 버전으로 복귀 가능

## 3. 4가지 수정 수단 사용 가이드

| 수단 | 적합한 경우 | 비유 | 영상 시연 시점 |
|------|------------|------|----------------|
| 채팅 (Chat / Inline Comment) | 광범위 변경, 모호한 의도 | "디자이너에게 메시지" | (전반적) |
| **Tweaks** (슬라이더) | 같은 디자인의 변형 / 미세조정 | "Figma Variants + 자동 슬라이더" | 12:00~12:48 |
| **Drawing** (영역 마킹) | 영역 단위 부분 수정 | "포토샵 선택 영역 + 코멘트" | 12:50~13:08 |
| **Edit** (요소 직접) | 픽셀/속성 단위 정확 변경 | "Figma 우측 속성 패널" | 13:10~13:31 |

### 추천 사용 흐름

```
초안 생성  →  큰 방향 수정 (채팅)  →  변형 탐색 (Tweaks)  →  영역 수정 (Drawing)  →  마감 (Edit)
```

### Tweaks의 차별 포인트

- 일반 슬라이더는 **고정 속성**(투명도, 회전 등)을 조정한다.
- Claude Design의 Tweaks는 **Claude가 컨텍스트에 맞춰 슬라이더 자체를 디자인**한다.
  - 예: 슬라이드덱이라면 "쉐이더 강도", "신 지속 시간", "액센트 컬러"가 자동으로 슬라이더화됨.
  - 이는 디자인 의도가 슬라이더 정의에 반영되었다는 의미다.

### Drawing의 숨은 효과

영상 사례에서 **하나의 영역만 수정 요청했는데 같은 패턴이 적용된 다른 요소까지 일괄 수정**되었다(자막 13:03~13:08). Claude가 **시각적 패턴을 인식**하고 일관성 있게 적용한다는 의미다.

## 4. 디자인 시스템 활용 팁

### 추천 워크플로우 (영상 강력 추천)

> Figma Community → 글로벌 기업 디자인 시스템 다운로드 → Claude Design에 임포트 → 즉시 키스크린 자동 생성

#### Figma Community 추천 디자인 시스템

| 시스템 | 제작 | 특징 |
|--------|------|------|
| **Carbon Design System** | IBM | 방대한 컴포넌트, 강력한 접근성, "교과서 같은 디자인 시스템" — 영상 발표자 추천 |
| **Material Design 3** | Google | 광범위한 모바일 패턴 |
| **Atlassian Design System** | Atlassian | 엔터프라이즈 SaaS 패턴 |
| **Polaris** | Shopify | 커머스 패턴 |

### design.md 라이브러리

GitHub의 **awesome-design-md** 레포에는 글로벌 브랜드의 design.md 파일이 정리되어 있다 (영상 자막 04:58 인용):

- Apple, BMW, IBM, Coinbase, Spotify 등

> 외부 GitHub URL을 입력했을 때 접근 실패 사례가 영상에서 확인되므로, **노트 필드에 design.md 내용을 직접 붙여넣는 방식**을 권장한다.

### Need Review 단계 활용

디자인 시스템 생성 후 **Need Review N개 항목**이 카드로 표시된다:

- **Looks Good** / **Needs Work** 중 선택
- Needs Work 시 **수정 방향을 프롬프트로 작성** → Claude가 해당 항목만 재생성
- **꼼꼼히 거치면 추후 재생성 횟수가 줄어들어 비용도 절감된다.**

### Publish vs Default 토글

| 토글 | 의미 |
|------|------|
| Publish | 디자인 시스템을 활성화 (사용 가능 상태) |
| Default | 새 프로젝트 생성 시 자동 적용되는 기본 시스템 |

여러 디자인 시스템을 함께 관리할 때는 **Default를 의도적으로 비활성화**하고 프로젝트별로 명시적 선택을 권장한다.

## 5. 외부 컴포넌트 코드 첨부 패턴

영상 17:43~18:35 사례:

- **3D 글로브 프로토타입** 1차 시도(프롬프트만): 평범한 퀄리티
- **react-globe.gl** 컴포넌트 코드를 GitHub에서 복사 → 첨부 → 퀄리티 급상승

### 시사점

- Claude Design은 **외부 라이브러리 코드를 컨텍스트로 받으면 결과 퀄리티가 크게 향상**된다.
- 특정 컴포넌트(차트, 글로브, 셰이더 등)를 사용하고 싶다면 **해당 라이브러리의 핵심 코드를 함께 제공**하는 것이 효과적이다.

## 6. 알려진 한계 및 우회 방법

### 6.1 인라인 코멘트 간헐적 소실

| 증상 | 코멘트가 일시적으로 사라지거나 적용되지 않음 |
| 우회 | **채팅 패널에 동일한 요청을 다시 입력** |

### 6.2 컴팩트 뷰 저장 오류

| 증상 | 컴팩트 뷰 모드에서 저장 시 실패 발생 |
| 우회 | **일반 뷰로 전환 후 저장** |

### 6.3 대규모 코드베이스 연결 시 지연

| 증상 | 디자인 시스템 자동 학습이 매우 느려짐 |
| 우회 | **핵심 모듈/디렉토리만 선택해 업로드** |

### 6.4 외부 GitHub URL 접근 실패

| 증상 | GitHub URL 입력 시 외부 레포에 접근 못 하는 경우 |
| 우회 | **노트 필드에 design.md 내용을 직접 붙여넣기** |

### 6.5 데스크탑 앱 미지원

| 증상 | Claude 데스크탑 앱에서는 Design 메뉴 자체가 보이지 않음 |
| 우회 | **웹 브라우저에서 Claude에 로그인** |

### 6.6 레이아웃 부분 깨짐

| 증상 | 일부 결과물에서 레이아웃이 어긋나는 경우 발생 |
| 우회 | **Drawing 또는 Edit으로 해당 영역만 부분 수정** |

## 7. 비용 가이드

### 실측 사례 (영상 자막 19:11)

> 발표자 진술: "이번에 테스트하고 시연하는 과정에서만 따로 **47달러 정도가 추가**로 들었습니다."

### 비용 절감 패턴

| 패턴 | 효과 |
|------|------|
| 디자인 시스템은 **한 번만 잘 구축** | Need Review를 꼼꼼히 거쳐 재생성 횟수 최소화 |
| 변형은 **Tweaks로** | 매번 재생성하지 않고 슬라이더로 즉시 변형 |
| **Wireframe 단계로 컨셉 확정** 후 **Hi-Fidelity 1회 생성** | 고비용 단계 횟수 최소화 |
| **외부 컴포넌트 코드 첨부**로 1회 시도 안에 좋은 결과 | 재시도 절약 |
| 디자인 시스템·자산을 **재사용** | 매번 처음부터 만들지 않음 |

> Pro/Max 플랜 기본 토큰 외 추가 사용량이 별도 과금되는 구조로 추정되므로, 본격 도입 전 **테스트 세션의 실제 청구액을 확인**한 후 도입 의사결정을 하는 것을 권장한다.

## 8. 적합한 시나리오 vs 부적합 시나리오

### 적합

- **빠른 컨셉 시각화** (와이어프레임 → 하이파이)
- 기존 **디자인 시스템에 따른 일관된 결과물 양산** (슬라이드, 마케팅 페이지)
- **Figma 임포트로 키스크린 빠르게 만들기**
- **피치덱 / 발표 자료**
- **카드 뉴스 / 소셜 자산**
- **셰이더/3D/애니메이션 프로토타입**

### 부적합

- 픽셀 단위 정확도가 필요한 **인쇄물 디자인**
- **장기 유지보수가 필요한 본격 디자인 SoT** (Figma 등 전통 도구가 더 적합)
- **저작권/규정 민감 결과물** (자동 생성 자산 사용 시 라이선스 검토 필요)
- 비용에 매우 민감한 **반복 작업** (재생성을 자주 한다면 Pro 토큰 한도를 빠르게 소진)

## 9. 실전 체크리스트

- [ ] 시작 전: 만들 결과물의 **목표/레이아웃/콘텐츠/대상** 4요소를 한 줄씩 메모
- [ ] 시작 전: 사용할 **디자인 시스템 결정** (회사 / Figma Community / design.md 라이브러리)
- [ ] 시작 후: 1차 프롬프트는 **콘텐츠 + 디자인 시스템**까지만, 비주얼 개선은 2차로 분리
- [ ] 수정: **채팅 → Tweaks → Drawing → Edit** 순으로 단계적 적용
- [ ] 마감: 익스포트 형식 결정 (Canva / PDF / PPTX / HTML / Claude Code)
- [ ] 마감: 비용 청구 확인
- [ ] 협업: 조직 내 URL 권한 적절히 설정 (뷰/편집)
