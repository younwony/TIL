# DevOps란?

> `[1] 정의` · 선수 지식: 없음

> 개발(Development)과 운영(Operations)을 통합하여 소프트웨어 개발 라이프사이클을 단축하고 품질을 향상시키는 문화·방법론·도구의 집합

`#DevOps` `#데브옵스` `#Development` `#Operations` `#CI` `#CD` `#ContinuousIntegration` `#ContinuousDelivery` `#ContinuousDeployment` `#자동화` `#Automation` `#InfrastructureAsCode` `#IaC` `#모니터링` `#Monitoring` `#피드백루프` `#FeedbackLoop` `#협업` `#Collaboration` `#애자일` `#Agile` `#마이크로서비스` `#컨테이너` `#Docker` `#Kubernetes` `#GitOps` `#SRE` `#DevSecOps` `#릴리즈` `#배포`

## 왜 알아야 하는가?

- **실무**: 현대 소프트웨어 개발에서 DevOps는 선택이 아닌 필수. 빠른 배포 주기와 안정적인 서비스 운영을 위해 DevOps 문화와 도구 이해가 필수적
- **면접**: "DevOps가 무엇인가?", "CI/CD 경험", "배포 자동화 경험" 등 DevOps 관련 질문이 빈출
- **기반 지식**: CI/CD, IaC, 모니터링, 컨테이너 등 모든 DevOps 실천법의 기초가 되는 개념

## 핵심 개념

- **문화 변화**: 개발팀과 운영팀 간의 벽을 허물고 협업하는 문화
- **자동화**: 빌드, 테스트, 배포, 인프라 관리의 자동화
- **지속적 피드백**: 모니터링과 로깅을 통한 빠른 피드백 루프
- **지속적 개선**: 작은 변경을 자주 배포하고 점진적으로 개선

## 쉽게 이해하기

**공장 비유**

전통적인 소프트웨어 개발은 **분리된 공장**과 같습니다:
- 설계팀(개발)이 제품을 만들면
- 생산팀(운영)에게 "이거 생산해" 하고 던짐
- 문제가 생기면 서로 책임 전가 ("우리 코드는 잘 돌아가는데요?")

DevOps는 **통합된 공장**입니다:
- 설계부터 생산까지 한 팀이 책임
- 자동화된 생산 라인(CI/CD)으로 빠른 출하
- 실시간 품질 모니터링으로 즉시 개선

```
┌─────────────────────────────────────────────────────────────────┐
│                    전통적 개발 vs DevOps                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   [전통적 개발]                                                  │
│   ┌──────────┐      ┌──────────┐      ┌──────────┐             │
│   │   개발   │ ──→  │   QA    │ ──→  │   운영   │             │
│   └──────────┘      └──────────┘      └──────────┘             │
│        ↑                                    │                   │
│        └────────── 느린 피드백 ─────────────┘                   │
│                   (주/월 단위)                                   │
│                                                                  │
│   [DevOps]                                                       │
│   ┌─────────────────────────────────────────────────┐           │
│   │              통합된 DevOps 팀                    │           │
│   │  ┌──────┐ → ┌──────┐ → ┌──────┐ → ┌──────┐    │           │
│   │  │ 코드 │   │ 빌드 │   │ 테스트│   │ 배포 │    │           │
│   │  └──────┘   └──────┘   └──────┘   └──────┘    │           │
│   │       ↑                                 │       │           │
│   │       └──────── 빠른 피드백 ────────────┘       │           │

│   └─────────────────────────────────────────────────┘           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 상세 설명

### DevOps의 탄생 배경

전통적인 소프트웨어 개발에서는 개발팀과 운영팀이 분리되어 있었습니다:

| 구분 | 개발팀 | 운영팀 |
|------|--------|--------|
| 목표 | 새로운 기능 빠르게 출시 | 시스템 안정성 유지 |
| 관점 | 변화 = 가치 | 변화 = 위험 |
| 문제 | "내 PC에서는 됐는데..." | "왜 이런 코드를 배포했지?" |

이러한 사일로(Silo)는 다음 문제를 야기했습니다:
- 배포 주기가 길어짐 (월/분기 단위)
- 배포 시 장애 발생 빈도 증가
- 책임 전가와 비효율적인 협업

**왜 이렇게 하는가?**

DevOps는 이 문제를 해결하기 위해 등장했습니다. "You build it, you run it" 원칙으로 개발팀이 운영까지 책임지면, 운영 관점을 고려한 더 안정적인 코드를 작성하게 됩니다.

### DevOps의 핵심 원칙 (CALMS)

DevOps의 핵심을 설명하는 CALMS 프레임워크:

| 원칙 | 영문 | 설명 |
|------|------|------|
| **문화** | Culture | 협업, 신뢰, 실패에서 배우는 문화 |
| **자동화** | Automation | 반복 작업의 자동화 (CI/CD, IaC) |
| **린** | Lean | 낭비 제거, 가치 흐름 최적화 |
| **측정** | Measurement | 메트릭 기반 의사결정 |
| **공유** | Sharing | 지식과 책임의 공유 |

### DevOps 라이프사이클 (무한 루프)

DevOps는 무한 루프(∞)로 표현됩니다:

```
        ┌─────────────────────────────────────────────┐
        │              DevOps 무한 루프               │
        ├─────────────────────────────────────────────┤
        │                                             │
        │          Plan → Code → Build → Test        │
        │         ↗                          ↘        │
        │      Monitor                     Release    │
        │         ↖                          ↙        │
        │          Operate ← Deploy ←────────         │
        │                                             │
        │   ┌─────────────────────────────────────┐   │
        │   │  Dev 영역        │    Ops 영역      │   │
        │   │  Plan, Code     │  Deploy, Operate │   │
        │   │  Build, Test    │  Monitor         │   │
        │   │  Release        │                  │   │
        │   └─────────────────────────────────────┘   │
        │                                             │
        └─────────────────────────────────────────────┘
```

| 단계 | 설명 | 주요 도구 |
|------|------|----------|
| **Plan** | 요구사항 정의, 백로그 관리 | Jira, Trello, GitHub Issues |
| **Code** | 코드 작성, 버전 관리 | Git, GitHub, GitLab |
| **Build** | 컴파일, 빌드 | Maven, Gradle, npm |
| **Test** | 자동화 테스트 | JUnit, Jest, Selenium |
| **Release** | 릴리즈 준비, 승인 | GitHub Releases, ArgoCD |
| **Deploy** | 배포 자동화 | Kubernetes, Ansible, Terraform |
| **Operate** | 인프라 운영, 설정 관리 | Docker, Kubernetes |
| **Monitor** | 로깅, 메트릭, 알림 | Prometheus, Grafana, ELK |

### DevOps 핵심 실천법

#### 1. CI/CD (지속적 통합/배포)

```
코드 푸시 → 자동 빌드 → 자동 테스트 → 자동 배포
```

- **CI (Continuous Integration)**: 코드 변경을 자주 통합하고 자동으로 빌드/테스트
- **CD (Continuous Delivery)**: 언제든 배포 가능한 상태 유지
- **CD (Continuous Deployment)**: 자동으로 프로덕션 배포

#### 2. IaC (Infrastructure as Code)

인프라를 코드로 정의하고 버전 관리:

```yaml
# Terraform 예시
resource "aws_instance" "web" {
  ami           = "ami-12345678"
  instance_type = "t2.micro"

  tags = {
    Name = "web-server"
  }
}
```

**장점:**
- 인프라 변경 이력 추적
- 동일한 환경 재현 가능
- 코드 리뷰를 통한 인프라 검증

#### 3. 모니터링과 관찰 가능성

| 구성 요소 | 설명 | 도구 예시 |
|----------|------|----------|
| **로깅** | 이벤트/에러 기록 | ELK, Loki |
| **메트릭** | 수치 데이터 수집 | Prometheus, CloudWatch |
| **트레이싱** | 요청 흐름 추적 | Jaeger, Zipkin |
| **알림** | 이상 징후 알림 | PagerDuty, Slack |

### DevOps 핵심 메트릭 (DORA)

DORA(DevOps Research and Assessment) 4대 핵심 메트릭:

| 메트릭 | 설명 | Elite 기준 |
|--------|------|-----------|
| **배포 빈도** | 얼마나 자주 배포하는가? | 하루 여러 번 |
| **변경 리드 타임** | 코드 커밋부터 배포까지 시간 | 1시간 이내 |
| **변경 실패율** | 배포 후 롤백/핫픽스 비율 | 15% 미만 |
| **복구 시간 (MTTR)** | 장애 발생부터 복구까지 시간 | 1시간 이내 |

### DevOps vs 관련 개념

| 개념 | 설명 | DevOps와의 관계 |
|------|------|----------------|
| **Agile** | 반복적, 점진적 개발 방법론 | DevOps의 문화적 기반 |
| **SRE** | Site Reliability Engineering | DevOps의 구현 방법 중 하나 |
| **DevSecOps** | 보안을 DevOps에 통합 | DevOps + Security |
| **GitOps** | Git을 단일 진실의 원천으로 | DevOps 실천법 중 하나 |
| **Platform Engineering** | 개발자 플랫폼 제공 | DevOps의 발전 형태 |

## 트레이드오프

| 장점 | 단점 |
|------|------|
| 배포 속도 향상 | 초기 학습 곡선 |
| 시스템 안정성 향상 | 도구/인프라 투자 필요 |
| 팀 간 협업 개선 | 조직 문화 변화 필요 |
| 빠른 피드백 루프 | 보안 고려 복잡성 증가 |
| 자동화로 인적 오류 감소 | 자동화 구축 초기 비용 |

## 면접 예상 질문

### Q: DevOps란 무엇인가요?

A: DevOps는 Development(개발)와 Operations(운영)의 합성어로, 소프트웨어 개발과 운영을 통합하는 문화·방법론·도구의 집합입니다. 핵심은 개발팀과 운영팀 간의 협업, 자동화(CI/CD, IaC), 지속적인 피드백입니다. "You build it, you run it" 원칙으로 개발팀이 운영까지 책임지며, 이를 통해 배포 주기를 단축하고 시스템 안정성을 높입니다.

### Q: CI/CD가 무엇이고 왜 중요한가요?

A: CI(Continuous Integration)는 코드 변경을 자주 통합하고 자동으로 빌드/테스트하는 것이고, CD는 Continuous Delivery(언제든 배포 가능 상태 유지) 또는 Continuous Deployment(자동 프로덕션 배포)입니다. 중요한 이유는:
1. 통합 문제를 조기에 발견
2. 수동 작업으로 인한 인적 오류 감소
3. 배포 주기 단축으로 비즈니스 민첩성 향상
4. 일관된 품질 보장

### Q: DevOps 도입 시 가장 어려운 점은 무엇인가요?

A: 기술적 도구보다 **문화 변화**가 가장 어렵습니다. 개발팀과 운영팀 간의 사일로를 허물고, 실패를 비난하지 않고 학습 기회로 삼는 문화, 협업과 공유의 문화를 만들어야 합니다. 도구는 문화가 받쳐주지 않으면 효과가 없습니다. 또한 기존 레거시 시스템과의 통합, 보안 고려사항 증가도 도전 과제입니다.
