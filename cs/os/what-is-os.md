## 왜 알아야 하는가?

운영체제는 모든 소프트웨어가 실행되는 기반입니다. 프로세스, 스레드, 메모리, 파일 시스템 등 백엔드 개발의 핵심 개념이 모두 운영체제에서 출발합니다. 성능 최적화, 동시성 문제 해결, 시스템 장애 분석을 위해서는 운영체제의 동작 원리를 이해해야 합니다. 면접에서도 OS 지식은 필수로 요구됩니다.

## 핵심 개념

- **운영체제 (Operating System)**: 하드웨어와 사용자/애플리케이션 사이의 중재자 역할
- **커널 (Kernel)**: 운영체제의 핵심, 하드웨어를 직접 제어
- **시스템 콜 (System Call)**: 애플리케이션이 커널 기능을 요청하는 인터페이스
- **프로세스 관리**: CPU 시간을 여러 프로그램에 배분
- **메모리 관리**: RAM을 효율적으로 할당하고 보호

## 쉽게 이해하기

**운영체제**를 건물 관리인에 비유할 수 있습니다.

- **건물 = 컴퓨터 하드웨어**: CPU, 메모리, 디스크, 네트워크
- **입주자 = 애플리케이션**: 웹 브라우저, 게임, 서버 프로그램
- **관리인 = 운영체제**: 입주자들이 건물을 공정하게 사용하도록 관리

**관리인의 역할**:
- **방 배정 (메모리 관리)**: 입주자에게 공간 할당, 다른 방 침범 방지
- **시간 배분 (CPU 스케줄링)**: 공용 시설을 순번대로 사용하게 함
- **민원 처리 (시스템 콜)**: 입주자의 요청(전등 교체 등)을 처리
- **보안 관리 (권한 제어)**: 허가된 입주자만 특정 공간 접근

### 운영체제의 역할

```
┌─────────────────────────────────────────────────────────────┐
│                    사용자 (User)                             │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│              응용 프로그램 (Application)                      │
│         브라우저, 게임, 서버, IDE, 오피스 등                  │
└──────────────────────────┬──────────────────────────────────┘
                           │ 시스템 콜 (System Call)
┌──────────────────────────▼──────────────────────────────────┐
│                    운영체제 (OS)                             │
│  ┌────────────┬────────────┬────────────┬────────────┐      │
│  │ 프로세스   │   메모리   │  파일      │  장치      │      │
│  │   관리     │    관리    │  시스템    │  드라이버  │      │
│  └────────────┴────────────┴────────────┴────────────┘      │
│                         커널 (Kernel)                        │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                    하드웨어 (Hardware)                       │
│              CPU, RAM, Disk, Network, GPU 등                 │
└─────────────────────────────────────────────────────────────┘
```

### 운영체제의 4가지 핵심 기능

#### 1. 프로세스 관리 (Process Management)

**역할**: 프로그램 실행, CPU 시간 배분, 프로세스 간 통신

```
┌───────────────────────────────────────────────────┐
│                    CPU (1개)                      │
│   시간 분할 → 프로세스 A, B, C가 번갈아 실행      │
└───────────────────────────────────────────────────┘
        │
        ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│프로세스A│  │프로세스B│  │프로세스C│
│ (실행)  │  │ (대기)  │  │ (대기)  │
└─────────┘  └─────────┘  └─────────┘
   10ms        10ms         10ms    → 각각 차례대로
```

**왜 필요한가?**
- CPU는 한 번에 하나의 명령만 실행 가능
- 여러 프로그램이 동시에 실행되는 것처럼 보이게 함 (멀티태스킹)
- 프로세스 간 간섭 방지 (격리)

#### 2. 메모리 관리 (Memory Management)

**역할**: RAM 할당/해제, 가상 메모리, 메모리 보호

```
┌─────────────────────────────────────┐
│           물리 메모리 (RAM)          │
├─────────┬─────────┬─────────┬───────┤
│  OS     │프로세스A│프로세스B│ 빈공간│
│ 영역    │  영역   │  영역   │       │
└─────────┴─────────┴─────────┴───────┘
```

**왜 필요한가?**
- 프로그램마다 필요한 메모리 양이 다름
- 프로세스가 다른 프로세스의 메모리를 침범하면 안 됨
- RAM보다 큰 프로그램도 실행 가능하게 (가상 메모리)

#### 3. 파일 시스템 관리 (File System Management)

**역할**: 파일/디렉터리 생성, 읽기/쓰기, 권한 관리

```
/
├── home/
│   └── user/
│       ├── documents/
│       └── downloads/
├── etc/
│   └── config.txt
└── var/
    └── log/
```

**왜 필요한가?**
- 디스크의 물리적 위치를 사람이 이해하기 쉬운 이름으로 추상화
- 파일 접근 권한 관리 (누가 읽고/쓸 수 있는지)
- 데이터 무결성 보장 (저널링)

#### 4. 장치 관리 (Device Management)

**역할**: 하드웨어 장치 제어, 드라이버 관리, I/O 처리

```
애플리케이션 → "파일 읽어줘"
      ↓
운영체제 → 디스크 드라이버에 명령
      ↓
디스크 드라이버 → 하드웨어 제어
      ↓
하드디스크 → 데이터 반환
```

**왜 필요한가?**
- 다양한 하드웨어를 일관된 인터페이스로 사용
- 애플리케이션이 하드웨어 세부사항을 몰라도 됨

### 커널 (Kernel)

**정의**: 운영체제의 핵심, 하드웨어를 직접 제어하는 소프트웨어

```
┌─────────────────────────────────────────────────┐
│                  사용자 공간                     │
│        (애플리케이션, 라이브러리)                │
├─────────────────────────────────────────────────┤
│                  커널 공간                       │
│   (프로세스 관리, 메모리 관리, 드라이버 등)      │
└─────────────────────────────────────────────────┘
```

**커널 모드 vs 사용자 모드**:

| 항목 | 사용자 모드 | 커널 모드 |
|------|------------|----------|
| 권한 | 제한적 | 모든 권한 |
| 하드웨어 접근 | 불가 | 가능 |
| 다른 프로세스 메모리 | 불가 | 가능 |
| 실행 주체 | 애플리케이션 | 커널 |

**왜 분리하는가?**
- 보안: 악성 프로그램이 시스템을 망가뜨리는 것 방지
- 안정성: 애플리케이션 오류가 전체 시스템에 영향 안 줌

### 시스템 콜 (System Call)

**정의**: 애플리케이션이 커널 기능을 요청하는 인터페이스

```
애플리케이션                    커널
     │                          │
     │  1. 시스템 콜 호출       │
     │  (예: read, write)       │
     ├─────────────────────────→│
     │                          │ 2. 커널 모드로 전환
     │                          │ 3. 요청 처리
     │  4. 결과 반환            │
     │←─────────────────────────┤
     │                          │
```

**주요 시스템 콜**:

| 카테고리 | 시스템 콜 | 설명 |
|---------|----------|------|
| 프로세스 | fork() | 새 프로세스 생성 |
| | exec() | 프로그램 실행 |
| | exit() | 프로세스 종료 |
| 파일 | open() | 파일 열기 |
| | read() | 파일 읽기 |
| | write() | 파일 쓰기 |
| | close() | 파일 닫기 |
| 메모리 | mmap() | 메모리 매핑 |
| | brk() | 힙 크기 조정 |
| 네트워크 | socket() | 소켓 생성 |
| | connect() | 연결 |
| | send()/recv() | 데이터 송수신 |

**왜 시스템 콜이 필요한가?**
- 하드웨어 접근은 커널만 가능
- 애플리케이션은 시스템 콜을 통해 간접적으로 요청
- 커널이 요청을 검증하고 안전하게 처리

### 운영체제의 종류

| OS | 특징 | 사용처 |
|-----|------|--------|
| **Linux** | 오픈소스, 서버 표준, 다양한 배포판 | 서버, 클라우드, 임베디드 |
| **Windows** | GUI 중심, 상용, 호환성 | 데스크톱, 기업 환경 |
| **macOS** | Unix 기반, Apple 생태계 | 개발, 디자인 |
| **Unix** | 안정성, 다중 사용자 | 메인프레임, 서버 |
| **Android** | Linux 기반, 모바일 | 스마트폰, 태블릿 |
| **iOS** | Unix 기반, Apple 모바일 | iPhone, iPad |

### 부팅 과정 (Boot Process)

```
1. 전원 ON
      ↓
2. BIOS/UEFI 실행
   - 하드웨어 초기화
   - 부트 장치 탐색
      ↓
3. 부트로더 (GRUB 등)
   - 커널 로드
      ↓
4. 커널 초기화
   - 드라이버 로드
   - 파일시스템 마운트
      ↓
5. init/systemd 실행
   - 서비스 시작
      ↓
6. 로그인 화면/셸
```

## 면접 예상 질문

### Q: 운영체제의 역할은 무엇인가요?

A: 운영체제는 하드웨어를 관리하고 애플리케이션에 서비스를 제공합니다. **4가지 핵심 역할**: (1) 프로세스 관리: CPU 시간 배분, 멀티태스킹 (2) 메모리 관리: RAM 할당, 가상 메모리, 보호 (3) 파일 시스템: 파일/디렉터리 관리, 권한 (4) 장치 관리: 하드웨어 드라이버, I/O 처리. **왜 필요한가?** 애플리케이션이 하드웨어를 직접 제어하면 충돌과 보안 문제가 발생하므로, OS가 중재자 역할을 합니다.

### Q: 커널이란 무엇인가요?

A: 커널은 운영체제의 핵심으로, 하드웨어를 직접 제어하는 소프트웨어입니다. 프로세스 관리, 메모리 관리, 장치 드라이버 등을 담당합니다. **커널 모드**에서 실행되어 모든 하드웨어에 접근 가능하고, **사용자 모드**의 애플리케이션은 시스템 콜을 통해 커널 기능을 요청합니다. **왜 분리하는가?** 악성 프로그램이 시스템을 망가뜨리는 것을 방지하고, 애플리케이션 오류가 전체 시스템에 영향 주지 않도록 합니다.

### Q: 시스템 콜이란 무엇인가요?

A: 시스템 콜은 애플리케이션이 커널 기능을 요청하는 인터페이스입니다. **왜 필요한가?** 하드웨어 접근은 커널 모드에서만 가능하므로, 애플리케이션은 시스템 콜(fork, read, write 등)을 통해 간접적으로 요청합니다. **동작 과정**: (1) 애플리케이션이 시스템 콜 호출 (2) 커널 모드로 전환 (3) 커널이 요청 처리 (4) 결과 반환 후 사용자 모드로 복귀. **비용**: 모드 전환에 오버헤드가 있어 과도한 시스템 콜은 성능 저하를 유발합니다.

### Q: 커널 모드와 사용자 모드의 차이는?

A: **사용자 모드**는 애플리케이션이 실행되는 제한된 환경으로, 하드웨어 직접 접근이 불가하고 다른 프로세스 메모리에 접근할 수 없습니다. **커널 모드**는 OS 커널이 실행되는 환경으로, 모든 하드웨어와 메모리에 접근 가능합니다. **왜 분리하는가?** (1) 보안: 악성 프로그램이 시스템 파괴 방지 (2) 안정성: 애플리케이션 버그가 OS 전체를 죽이지 않음. **전환 비용**: 컨텍스트 저장/복원으로 수백~수천 CPU 사이클 소요.
