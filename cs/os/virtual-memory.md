# 가상 메모리 (Virtual Memory)

> `[3] 중급` · 선수 지식: [프로세스와 스레드](./process-vs-thread.md), [메모리 관리](./memory-management.md)

> 물리 메모리의 크기와 무관하게 각 프로세스에게 독립적이고 연속적인 주소 공간을 제공하는 메모리 추상화 기법

`#가상메모리` `#VirtualMemory` `#주소변환` `#AddressTranslation` `#MMU` `#MemoryManagementUnit` `#TLB` `#TranslationLookasideBuffer` `#페이지테이블` `#PageTable` `#페이지폴트` `#PageFault` `#요구페이징` `#DemandPaging` `#COW` `#CopyOnWrite` `#스왑` `#Swap` `#SwapSpace` `#mmap` `#MemoryMappedFile` `#가상주소` `#VirtualAddress` `#물리주소` `#PhysicalAddress` `#페이지` `#Page` `#프레임` `#Frame` `#다단계페이지테이블` `#MultiLevelPageTable` `#역페이지테이블` `#InvertedPageTable`

## 왜 알아야 하는가?

가상 메모리는 현대 운영체제의 가장 중요한 추상화 중 하나입니다. 8GB RAM을 가진 컴퓨터에서 수십 개의 프로그램이 동시에 실행되고, 각 프로그램이 마치 메모리를 독점하는 것처럼 동작하는 것이 가상 메모리 덕분입니다. OOM 에러 해결, JVM 튜닝, 컨테이너 메모리 설정 등 실무 문제를 이해하려면 이 개념이 필수입니다.

- **실무**: OOM 에러 해결, JVM 힙 설정, Docker 메모리 제한, mmap 활용
- **면접**: 가상 메모리 동작 원리, 페이지 폴트, TLB는 CS 면접 핵심 주제
- **기반 지식**: 프로세스 격리, 메모리 보호, 공유 라이브러리, 파일 시스템 캐시 이해

## 핵심 개념

- **가상 주소 공간**: 각 프로세스가 독립적으로 사용하는 논리적 메모리 공간
- **주소 변환 (Address Translation)**: 가상 주소 → 물리 주소 변환 (MMU가 수행)
- **페이지 테이블**: 가상 페이지 → 물리 프레임 매핑 정보 저장
- **TLB**: 자주 사용하는 주소 변환을 캐싱하는 하드웨어
- **요구 페이징 (Demand Paging)**: 실제로 접근할 때만 메모리에 적재하는 기법
- **Copy-on-Write (COW)**: 쓰기 발생 전까지 메모리 페이지를 공유하는 최적화

## 쉽게 이해하기

**가상 메모리**를 아파트 호수 시스템에 비유할 수 있습니다.

### 아파트 호수 = 가상 주소

모든 아파트는 "101호, 102호, 201호..."라는 호수 체계를 사용합니다. A아파트의 101호와 B아파트의 101호는 같은 번호이지만 **완전히 다른 물리적 공간**입니다.

- **가상 주소 = 호수**: 각 프로세스가 0번지부터 시작하는 동일한 주소 체계 사용
- **물리 주소 = GPS 좌표**: 실제 RAM의 위치
- **MMU = 관리사무소**: 호수를 실제 위치로 변환
- **TLB = 자주 가는 집 메모장**: 매번 관리사무소 안 가도 됨

```
프로세스 A의 0x1000 → 물리 0x5000
프로세스 B의 0x1000 → 물리 0x9000  ← 같은 가상 주소, 다른 물리 주소!
```

### 요구 페이징 = 필요할 때만 이사짐 풀기

새 아파트로 이사할 때 모든 짐을 한 번에 풀지 않습니다. 필요한 상자만 먼저 열고, 나머지는 창고(디스크)에 둡니다.

- **이사짐 풀기 = 페이지 적재**: 실제로 사용할 때만 메모리에 올림
- **"그 상자 어디?" = 페이지 폴트**: 필요한 데이터가 메모리에 없으면 디스크에서 가져옴

### Copy-on-Write = 공유 복사기

회사에서 같은 문서를 여러 팀이 읽기만 한다면 **한 부만** 있으면 됩니다. 누군가 수정하려 할 때 그 때 **복사본**을 만듭니다.

```
fork() 직후:
  부모 프로세스 ──→ [페이지 A] ←── 자식 프로세스  (공유, 읽기 전용)

자식이 페이지 A에 쓰기 시도:
  부모 프로세스 ──→ [페이지 A (원본)]
  자식 프로세스 ──→ [페이지 A' (복사본)]  ← 이때 복사 발생
```

## 상세 설명

### 가상 주소 공간 구조

```
┌──────────────────────┐ 0xFFFFFFFF (높은 주소)
│     커널 영역         │ ← 모든 프로세스가 공유 (직접 접근 불가)
├──────────────────────┤ 0xC0000000 (3GB, Linux 32비트 기준)
│     Stack            │ ← 아래로 성장 (↓)
│         │            │
│         ▼            │
│                      │
│     (빈 공간)        │ ← 가상 주소만 할당, 물리 메모리 미할당
│                      │
│         ▲            │
│         │            │
│     Heap             │ ← 위로 성장 (↑)
├──────────────────────┤
│     BSS (미초기화)    │
│     Data (초기화)     │
│     Code (Text)      │ ← 읽기 전용, 공유 가능
└──────────────────────┘ 0x00000000 (낮은 주소)
```

**왜 Stack과 Heap이 반대 방향으로 성장하는가?**
- 두 영역의 최대 크기를 미리 알 수 없음
- 반대 방향으로 성장하면 빈 공간을 유동적으로 활용 가능
- Stack이 많이 쓰이면 Heap 공간 양보, 반대도 가능

### 주소 변환 과정

#### 단일 레벨 페이지 테이블

```
가상 주소 (32비트):
┌────────────────────┬──────────────┐
│   페이지 번호 (20bit)│  오프셋 (12bit)│
└────────────────────┴──────────────┘
         │                    │
         ▼                    │
   ┌──────────┐               │
   │페이지 테이블│              │
   ├──────────┤               │
   │ VPN → PFN│               │
   │  0  → 5  │               │
   │  1  → 8  │               │
   │  2  → 2  │               │
   └──────────┘               │
         │                    │
         ▼                    │
   물리 프레임 번호            │
         │                    │
         ▼                    ▼
   ┌────────────────────┬──────────────┐
   │  프레임 번호 (20bit) │  오프셋 (12bit)│ → 물리 주소
   └────────────────────┴──────────────┘
```

**페이지 크기가 4KB (12bit 오프셋)인 이유**:
- 너무 작으면: 페이지 테이블 엔트리 수 증가 → 테이블 크기 폭증
- 너무 크면: 내부 단편화 증가 → 메모리 낭비
- 4KB는 두 요인의 균형점

**단일 레벨의 문제**:

```
32비트 주소 공간, 4KB 페이지:
  페이지 수 = 2^32 / 2^12 = 2^20 = 약 100만 개
  각 엔트리 4바이트 → 페이지 테이블 크기 = 4MB

프로세스마다 4MB → 100개 프로세스 = 400MB (페이지 테이블만!)
64비트에서는 더 심각
```

#### 다단계 페이지 테이블

```
가상 주소 (32비트, 2단계):
┌──────────┬──────────┬──────────────┐
│ L1 (10bit)│ L2 (10bit)│ 오프셋 (12bit)│
└──────────┴──────────┴──────────────┘
     │           │
     ▼           │
┌──────────┐     │
│ 1단계      │    │
│ 페이지 디렉│    │
│ 토리       │    │
│ [0] ──────┼────┤
│ [1] null  │    │
│ [2] null  │    │   ← 사용 안 하는 영역은 할당하지 않음!
│ ...       │    │
└──────────┘     │
                  ▼
            ┌──────────┐
            │ 2단계      │
            │ 페이지     │
            │ 테이블     │
            │ [0] → PFN │
            │ [1] → PFN │
            │ ...       │
            └──────────┘
```

**왜 다단계인가?**
- 사용하지 않는 주소 범위의 2단계 테이블을 생성하지 않음
- 단일 레벨: 항상 4MB (모든 엔트리 필요)
- 2단계: 실제 사용하는 영역만 테이블 생성 → 대폭 절약
- 64비트 시스템: 4~5단계 사용 (Linux는 5단계)

### TLB (Translation Lookaside Buffer)

```
CPU가 가상 주소 접근:

  ┌────────────┐     TLB 히트 (1~2ns)
  │ 가상 주소   │────────────────────────→ 물리 주소 → 메모리 접근
  └────────────┘
       │
       │ TLB 미스
       ▼
  ┌────────────┐
  │ 페이지 테이블 │─── 매핑 존재 ──→ TLB 갱신 → 물리 주소
  │ (메모리)    │
  └────────────┘
       │
       │ 매핑 없음 (valid=0)
       ▼
  ┌────────────┐
  │ 페이지 폴트  │─── 디스크에서 로드 → 페이지 테이블 갱신 → TLB 갱신
  │ (디스크 I/O)│
  └────────────┘
```

**각 단계의 비용**:

| 단계 | 소요 시간 | 빈도 |
|------|----------|------|
| TLB 히트 | ~1-2 ns | ~99% |
| TLB 미스 → 페이지 테이블 | ~10-100 ns | ~0.99% |
| 페이지 폴트 → 디스크 | ~1-10 ms | ~0.01% |

**왜 TLB 히트율이 99%인가?**

**지역성 원리 (Locality)**:
- **시간 지역성**: 최근 접근한 주소를 다시 접근할 확률이 높음 (반복문)
- **공간 지역성**: 인접 주소를 접근할 확률이 높음 (배열 순회)
- 하나의 TLB 엔트리가 4KB 페이지를 커버 → 인접 접근 모두 히트

### 요구 페이징 (Demand Paging)

프로그램 시작 시 모든 페이지를 메모리에 올리지 않고, **실제로 접근할 때만 적재**합니다.

```
프로그램 시작:
  모든 페이지의 valid bit = 0 (메모리에 없음)

실행 중 페이지 접근:
  1. valid bit 확인 → 0이면 페이지 폴트
  2. 디스크에서 해당 페이지 로드
  3. 빈 프레임에 적재
  4. 페이지 테이블 갱신 (valid = 1)
  5. 중단된 명령어 재실행
```

**왜 요구 페이징을 쓰는가?**
- **프로그램의 10-20%만 자주 사용** (Working Set)
- 나머지 80-90%는 에러 처리, 드물게 사용하는 기능
- 전부 올리면 메모리 낭비 + 시작 시간 증가

### Copy-on-Write (COW)

`fork()` 시스템 콜에서 핵심적인 최적화 기법입니다.

```java
// fork() 동작 (개념)
pid_t pid = fork();

if (pid == 0) {
    // 자식 프로세스
    // 부모의 메모리를 즉시 복사하지 않음 (COW)
    // 읽기만 하면 부모와 같은 물리 페이지 공유
    // 쓰기 시도 시 → 해당 페이지만 복사
}
```

**COW 동작 과정**:

```
[1] fork() 직후 - 모든 페이지를 읽기 전용으로 공유
    부모: [Page A (RO)] [Page B (RO)] [Page C (RO)]
              ↓               ↓               ↓
    물리:  [Frame 1]      [Frame 2]      [Frame 3]
              ↑               ↑               ↑
    자식: [Page A (RO)] [Page B (RO)] [Page C (RO)]

[2] 자식이 Page B에 쓰기 시도 → Page Fault!
    → Page B만 복사하여 새 프레임에 할당

    부모: [Page A (RO)] [Page B (RW)] [Page C (RO)]
              ↓               ↓               ↓
    물리:  [Frame 1]      [Frame 2]      [Frame 3]
              ↑                               ↑
    자식: [Page A (RO)] [Page B'(RW)] [Page C (RO)]
                              ↓
                         [Frame 4]  ← 새로 복사된 프레임
```

**왜 COW가 중요한가?**
- `fork()` 후 바로 `exec()`하는 패턴이 매우 흔함
- `exec()`은 메모리를 새 프로그램으로 완전 교체
- COW가 없으면 fork() 시 수백 MB를 복사한 뒤 즉시 폐기 → 낭비

### 메모리 매핑 파일 (mmap)

파일을 가상 메모리에 매핑하여 메모리처럼 접근합니다.

```
일반 파일 읽기:
  read() → 커널 버퍼 복사 → 유저 버퍼 복사 (2번 복사)

mmap:
  가상 주소 공간에 파일 매핑 → 직접 접근 (0번 복사)
```

```java
// Java NIO의 mmap 활용
FileChannel channel = FileChannel.open(Path.of("data.bin"), READ, WRITE);
MappedByteBuffer buffer = channel.map(FileChannel.MapMode.READ_WRITE, 0, channel.size());

// 파일을 메모리처럼 직접 접근
buffer.putInt(0, 42);       // offset 0에 정수 쓰기
int value = buffer.getInt(0); // offset 0에서 정수 읽기
```

**왜 mmap을 사용하는가?**
- 대용량 파일 처리 시 커널/유저 간 복사 제거
- 여러 프로세스가 같은 파일 매핑 → 공유 메모리 효과
- OS의 페이지 캐시 활용 → 자동 캐싱

## 트러블슈팅

### 사례 1: Java OOM과 가상 메모리

#### 증상

```
java.lang.OutOfMemoryError: Java heap space
```

#### 원인 분석

| 구분 | 가상 메모리 | 물리 메모리 | JVM 힙 |
|------|------------|------------|--------|
| 크기 | 수 TB (64비트) | 8-64 GB | -Xmx로 제한 |
| OOM 발생 | 거의 없음 | swap으로 확장 | 자주 발생 |

- 물리 메모리가 남아있어도 JVM 힙 제한에 걸리면 OOM
- 반대로, JVM 힙이 남아도 Native 메모리(Direct Buffer 등)가 부족하면 OOM

#### 해결 방법

```bash
# JVM 힙 크기 설정
java -Xms2g -Xmx4g -jar app.jar

# Native Memory Tracking
java -XX:NativeMemoryTracking=summary -jar app.jar

# 메모리 사용 현황 확인
jcmd <PID> VM.native_memory summary
```

### 사례 2: Docker 컨테이너 메모리 제한과 OOM Killer

#### 증상

```
컨테이너가 갑자기 종료됨
dmesg: Out of memory: Kill process <PID> (java)
```

#### 원인 분석

Docker는 cgroups로 메모리를 제한합니다. 프로세스가 제한을 초과하면 Linux OOM Killer가 프로세스를 강제 종료합니다.

```bash
# Docker 메모리 제한 설정
docker run -m 2g --memory-swap 2g my-app

# JVM은 컨테이너 메모리 제한을 인식해야 함
docker run -m 2g my-java-app java \
  -XX:MaxRAMPercentage=75.0 \
  -jar app.jar
```

#### 예방 조치

- JVM 힙을 컨테이너 메모리의 70~80%로 설정
- `-XX:MaxRAMPercentage` 사용으로 자동 비율 계산
- Native 메모리(스레드 스택, Direct Buffer, Metaspace)도 고려

## 예제 코드

### 가상 메모리 사용량 확인 (Linux)

```bash
# 프로세스별 가상 메모리 vs 물리 메모리
ps aux --sort=-%mem | head -10
#  VSZ: 가상 메모리 크기
#  RSS: 실제 물리 메모리 사용량 (Resident Set Size)

# 상세 메모리 맵
cat /proc/<PID>/maps

# 메모리 요약
cat /proc/<PID>/status | grep -E "VmSize|VmRSS|VmSwap"
# VmSize: 가상 메모리 총 크기
# VmRSS: 물리 메모리 사용량
# VmSwap: 스왑 사용량
```

### TLB 영향 확인 (Java)

```java
/**
 * 배열 접근 패턴에 따른 TLB 영향 시연
 * 순차 접근 vs 랜덤 접근 성능 비교
 */
public class TlbDemo {
    private static final int SIZE = 64 * 1024 * 1024; // 64MB
    private static final int[] array = new int[SIZE / 4];

    public static void main(String[] args) {
        // 순차 접근: TLB 히트율 높음 (공간 지역성)
        long start = System.nanoTime();
        long sum = 0;
        for (int i = 0; i < array.length; i++) {
            sum += array[i];
        }
        long sequential = System.nanoTime() - start;

        // 페이지 크기(4KB) 단위 점프 접근: TLB 미스 빈번
        start = System.nanoTime();
        sum = 0;
        int stride = 4096 / 4; // 4KB = 1024 int
        for (int i = 0; i < array.length; i += stride) {
            sum += array[i];
        }
        long strided = System.nanoTime() - start;

        System.out.println("순차 접근: " + sequential / 1_000_000 + "ms");
        System.out.println("페이지 점프 접근: " + strided / 1_000_000 + "ms");
    }
}
```

### Huge Pages 설정

```bash
# Linux에서 Huge Pages (2MB) 활성화
echo 512 > /proc/sys/vm/nr_hugepages  # 512 * 2MB = 1GB

# JVM에서 Huge Pages 사용
java -XX:+UseLargePages -Xmx4g -jar app.jar
```

**왜 Huge Pages인가?**
- 일반 4KB 페이지: 4GB 메모리 = 100만 페이지 = TLB 부족
- 2MB Huge Pages: 4GB 메모리 = 2048 페이지 = TLB 커버 가능
- DB, 캐시 서버 등 대용량 메모리 사용 시 TLB 미스 대폭 감소

## 트레이드오프

| 관점 | 장점 | 단점 |
|------|------|------|
| **메모리 추상화** | 프로세스 격리, 보안, 편리한 프로그래밍 | 주소 변환 오버헤드 |
| **요구 페이징** | 메모리 절약, 빠른 시작 | 페이지 폴트 지연 |
| **COW** | fork() 효율화, 메모리 절약 | 쓰기 시 복사 오버헤드 |
| **mmap** | 대용량 파일 효율, 공유 용이 | 복잡한 동기화, 비정상 종료 시 데이터 손실 |
| **Huge Pages** | TLB 효율 향상, 성능 개선 | 내부 단편화 증가, 설정 복잡 |

### 일반 페이지 vs Huge Pages

| 항목 | 4KB 페이지 | 2MB Huge Pages |
|------|-----------|----------------|
| TLB 커버리지 | 낮음 | 높음 (512배) |
| 내부 단편화 | 적음 | 많음 |
| 페이지 테이블 크기 | 큼 | 작음 |
| 적합한 경우 | 범용 | DB, 캐시, 대용량 메모리 |

## 면접 예상 질문

### Q: 가상 메모리란 무엇이고, 왜 사용하나요?

A: 가상 메모리는 각 프로세스에게 독립적인 연속 주소 공간을 제공하는 메모리 추상화입니다. **왜 사용하는가?** (1) **메모리 보호**: 프로세스 A가 B의 메모리를 접근할 수 없음. (2) **메모리 추상화**: 각 프로세스가 0번지부터 시작하는 전체 메모리를 독점하는 것처럼 사용. (3) **메모리 확장**: 물리 메모리보다 큰 프로그램 실행 가능 (디스크를 백업으로). (4) **메모리 공유**: 공유 라이브러리를 한 번만 물리 메모리에 올리고 여러 프로세스가 공유.

### Q: 페이지 폴트가 발생하면 어떤 일이 일어나나요?

A: (1) CPU가 가상 주소를 접근하면 MMU가 페이지 테이블의 valid bit를 확인합니다. (2) valid=0이면 **페이지 폴트 트랩**이 발생합니다. (3) OS가 해당 접근이 정당한지 확인합니다 (불법이면 Segmentation Fault). (4) 빈 프레임을 찾아 디스크에서 페이지를 로드합니다. 빈 프레임이 없으면 **페이지 교체 알고리즘**으로 victim을 선택합니다. (5) 페이지 테이블을 갱신하고 중단된 명령어를 재실행합니다. **왜 비싼가?** 디스크 접근은 메모리 접근보다 10만~100만 배 느립니다.

### Q: TLB가 무엇이고 왜 중요한가요?

A: TLB는 최근 사용한 가상→물리 주소 변환을 캐싱하는 **하드웨어 캐시**입니다. **왜 중요한가?** 가상 메모리를 사용하면 모든 메모리 접근마다 페이지 테이블 조회가 필요합니다. 다단계 페이지 테이블이면 3~5번 메모리를 추가 접근해야 합니다. TLB가 이를 캐싱하여 1회 접근으로 줄입니다. **TLB 히트율이 99%+**인 이유는 지역성 원리 덕분입니다. **프로세스 전환 시 TLB 플러시**가 비싼 이유도 여기 있습니다.

### Q: Copy-on-Write를 설명해주세요.

A: fork() 시 자식 프로세스의 메모리를 즉시 복사하지 않고, **부모와 같은 물리 페이지를 공유**하면서 **읽기 전용**으로 표시합니다. 부모/자식 중 하나가 **쓰기를 시도**하면 그때 해당 페이지만 복사합니다. **왜 COW를 쓰는가?** fork() 후 exec()하는 패턴이 매우 흔한데, exec()은 메모리를 완전히 교체합니다. COW가 없으면 fork()에서 수백 MB를 복사한 뒤 즉시 폐기하는 낭비가 발생합니다.

### Q: VSZ와 RSS의 차이는?

A: **VSZ (Virtual Size)**: 프로세스가 할당받은 가상 메모리 총 크기입니다. 실제로 물리 메모리에 올라가지 않은 영역도 포함합니다. **RSS (Resident Set Size)**: 실제로 물리 메모리에 올라가 있는 크기입니다. **VSZ > RSS인 이유**: 요구 페이징으로 접근하지 않은 페이지는 물리 메모리에 없고, 스왑 아웃된 페이지도 RSS에 포함되지 않습니다. **실무 의미**: 메모리 사용량 모니터링 시 **RSS가 실질적 사용량**입니다.
