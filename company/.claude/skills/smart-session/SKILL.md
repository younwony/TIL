# Smart Session — 통합 워크플로우 오케스트레이터

치트시트의 9가지 실전 기법을 **3단계 워크플로우**로 오케스트레이션하는 스킬.
기존 모듈(Plan Mode, Sub Agent, Hook, /handoff, /team-review 등)을 조합하여
세션 전체 라이프사이클을 관리한다.

## 트리거 키워드

- "스마트 세션", "세션 시작", "작업 시작" → Phase 1 실행
- "세션 정리", "세션 종료" → Phase 3 실행
- `/smart-session` → Phase 1부터 순차 실행

## Phase 1: 세션 시작 (컨텍스트 진단)

세션 시작 시 아래를 **자동 수행**한다.

### 1-1. 이전 맥락 로드

```
우선순위:
1. HANDOFF.md 존재 → 읽고 이전 작업 맥락 파악
2. TODO.md 존재 → 미완료 항목 확인
3. MEMORY.md 존재 → 관련 메모리 참조
4. 모두 없음 → 새 세션으로 시작
```

- HANDOFF.md가 있으면 내용을 요약하여 사용자에게 보고
- TODO.md가 있으면 미완료 항목 목록 표시
- 파일 읽기 후 사용자에게 "이전 작업을 이어서 할까요?" 확인

### 1-2. 세션 목표 단일화

- **한 세션 = 한 피처** 원칙 적용
- 사용자에게 이번 세션의 목표를 확인
- 복수 목표 요청 시 우선순위 정하여 하나씩 처리 제안

### 1-3. MCP/컨텍스트 상태 확인

- 연결된 MCP 서버 상태 간략 확인
- 불필요한 MCP가 토큰을 소비하고 있다면 사용자에게 알림

### Phase 1 출력 형식

```markdown
## 🔍 세션 진단 결과

**이전 맥락**: {HANDOFF.md 요약 또는 "새 세션"}
**미완료 작업**: {TODO.md 미완료 항목 수 또는 "없음"}
**세션 목표**: {사용자 확인 후 기록}
**MCP 상태**: {활성 MCP 목록}

---
준비 완료. 작업을 시작합니다.
```

## Phase 2: 작업 실행 (WAT 오케스트레이션)

작업 실행 중 아래 규칙을 **자동 적용**한다.
이 Phase는 별도 트리거 없이 세션 진행 중 상시 적용되는 규칙이다.

### 2-1. WAT 프레임워크 순서

```
[T] Tools  → MCP/Hook 상태 확인, 필요 도구 준비
[W] Workflows → Plan Mode로 설계 우선, 점진적 복잡도 증가
[A] Agents → 독립 작업은 Sub Agent에 병렬 위임
```

### 2-2. 점진적 복잡도 증가

- 작업을 단계별로 나누어 실행
- 각 단계 완료 후 테스트/검증 → 다음 단계 진행
- 한 번에 대규모 변경 금지

### 2-3. 에러 처리 원칙

- **에러는 원본 그대로**: 에러 로그/스택 트레이스는 가공 없이 전문 기반 분석
- 사용자가 에러를 보내면 원본 전체를 분석 대상으로 사용
- 에러 요약이나 부분 발췌 금지

### 2-4. 가정 변경 알림

- 기술 스택, 아키텍처, 라이브러리 선택 등 **가정이 변경**될 때
- 반드시 사용자에게 **먼저 확인** 후 진행
- 예: "기존에 MySQL을 사용하는 것으로 가정했는데, PostgreSQL로 변경하는 게 맞나요?"

### 2-5. 스크립트 오프로드

아래 상황 감지 시 셸 스크립트 생성을 **자동 제안**:
- 10개 이상 파일 일괄 변환/포맷팅
- 반복적인 텍스트 치환 작업
- 대량 데이터 변환/마이그레이션
- 빌드/배포 자동화 작업

```
제안 형식:
"이 작업은 {N}개 파일에 반복 적용됩니다.
셸 스크립트로 분리하면 더 효율적입니다. 스크립트를 생성할까요?"
```

### 2-6. work-plan-start 연동

- WORK-SPEC.md가 존재하면 `/work-plan-start` 팀 에이전트 워크플로우와 자동 연동
- Phase 2의 WAT 규칙이 work-plan-start 실행 중에도 적용됨

## Phase 3: 세션 종료 (상태 보존)

"세션 정리", "세션 종료" 키워드 또는 작업 완료 시 아래를 **자동 수행**한다.

### 3-1. 컨텍스트 상태 진단

```
응답 속도 기준:
- 정상 → 계속 진행
- 느려짐 감지 → "/compact 실행을 권장합니다" 제안
- 매우 느림/오류 → "/clear 후 새 세션 권장" 제안
```

### 3-2. 코드 리뷰 (코드 변경 시)

- 이번 세션에서 코드 변경이 있었다면:
  - 변경 규모 小 (1~2 파일) → 인라인 자체 리뷰
  - 변경 규모 中~大 (3+ 파일) → `/team-review` 실행 제안

### 3-3. TODO.md 업데이트

- 이번 세션에서 완료한 작업 항목을 체크
- 새로 발견된 작업이 있으면 추가
- TODO.md가 없으면 생성 제안

### 3-4. 주요 결정사항 메모리 저장

- 이번 세션에서 내린 주요 기술 결정 요약
- 사용자에게 메모리 저장 여부 확인
- 승인 시 적절한 메모리 타입(project/feedback/reference)으로 저장

### Phase 3 출력 형식

```markdown
## 📋 세션 정리 결과

**완료 작업**: {완료된 항목 목록}
**코드 변경**: {변경된 파일 수} 파일 ({추가/수정/삭제} 요약)
**미완료 작업**: {남은 항목 또는 "없음"}
**Git 상태**: {staging 상태 요약}

### 💡 제안
- {컨텍스트 상태에 따른 제안}
- {메모리 저장 제안}
- {리뷰 제안}
```

## 기존 모듈 활용 매핑

| 치트시트 기법 | 활용하는 기존 모듈 | Phase |
|-------------|------------------|-------|
| Second Brain | `/memory` (내장) | 1, 3 |
| Lazy Loading | `@파일명` 참조 | 1 |
| 한 세션=한 피처 | `/clear`, `/compact` | 1, 3 |
| MCP 모니터링 | MCP 상태 확인 | 1 |
| 스크립트 오프로드 | `Bash` 도구 | 2 |
| Thinking 로그 | Extended Thinking | 2 |
| 에러 원본 전달 | 자연어 규칙 | 2 |
| 크로스 리뷰 | `/team-review` | 3 |
| TODO.md | 파일 Write | 3 |
| WAT 프레임워크 | T→W→A 실행 순서 | 2 |
| 점진적 복잡도 | Plan Mode | 2 |
| 피드백 루프 | 테스트 실행 | 2 |

## 주의사항

- Phase 1, 3은 명시적 트리거 시에만 실행
- Phase 2 규칙은 세션 진행 중 상시 자동 적용
- 기존 개별 커맨드(`/handoff`, `/team-review` 등)는 독립적으로도 사용 가능
- smart-session은 이들을 조합하는 **상위 오케스트레이션 레이어**
