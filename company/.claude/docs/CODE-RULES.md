# 코드 작성 원칙 상세

> 이 문서는 CLAUDE.md에서 분리된 상세 코드 작성 규칙입니다.
> 코드 작업 시 필요에 따라 `@.claude/docs/CODE-RULES.md`로 참조하세요.

## 1. 테스트 코드 작성 (필수)

**모든 새로운 기능과 코드 변경에는 반드시 테스트 코드를 작성해야 합니다.**

### 테스트 작성 규칙

- **TDD (Test-Driven Development) 원칙 준수**
    - 테스트 작성 → 실패 확인 → 코드 작성 → 테스트 통과 → 리팩토링
    - 테스트를 먼저 작성하고 구현 코드를 작성할 것

- **테스트 커버리지**
    - 단위 테스트 (Unit Test): 모든 public 메서드에 대한 테스트 작성
    - 통합 테스트 (Integration Test): 주요 기능에 대한 통합 테스트 작성
    - 최소 80% 이상의 코드 커버리지 유지

- **테스트 코드 품질**
    - `@DisplayName`을 사용하여 테스트 의도를 명확하게 표현
    - 한글로 테스트 설명 작성하여 가독성 향상
    - Given-When-Then 패턴 사용 권장
    - 하나의 테스트는 하나의 기능만 검증

### 테스트 종류

1. **컨트롤러 테스트**
    - `@WebMvcTest` 사용
    - MockMvc를 통한 HTTP 요청/응답 검증
    - 모델 속성 및 뷰 이름 검증

2. **서비스 테스트**
    - 비즈니스 로직 검증
    - Mock 객체를 사용한 의존성 격리

3. **통합 테스트**
    - `@SpringBootTest` 사용
    - 전체 애플리케이션 컨텍스트 로드 검증
    - 실제 환경과 유사한 조건에서 테스트

## 2. 코드 품질

### 상수 사용
- 매직 넘버와 하드코딩된 문자열 사용 금지
- 모든 리터럴 값은 의미 있는 상수로 정의
- 상수는 `private static final`로 선언하고 대문자와 언더스코어 사용

### 코드 구조
- 단일 책임 원칙 (SRP) 준수
- 중복 코드 제거 (DRY 원칙)
- 메서드는 한 가지 일만 수행하도록 작성
- 적절한 메서드 분리로 가독성 향상

### 네이밍
- 명확하고 의미 있는 이름 사용
- 클래스명: PascalCase
- 메서드/변수명: camelCase
- 상수명: UPPER_SNAKE_CASE

## 3. 성능 최적화

- 불필요한 객체 생성 방지
- `DateTimeFormatter`, `Pattern` 등 재사용 가능한 객체는 상수로 선언
- 적절한 캐싱 전략 사용

## 4. 문서화

- 복잡한 로직에는 주석 작성
- Public API에는 JavaDoc 작성
- README.md 최신 상태 유지

## 5. 개발 워크플로우

1. **기능 개발 전**
    - 요구사항 명확히 파악
    - 테스트 케이스 먼저 작성 (TDD)

2. **개발 중**
    - 작은 단위로 작업
    - 변경사항은 git add만 진행
    - 테스트가 통과하는지 확인하며 개발

3. **개발 완료 후**
    - 전체 테스트 실행 및 통과 확인
    - git add로 staging 완료
    - 사용자가 commit 명령 시 commit 실행
    - 코드 리뷰 수행
    - 리팩토링 진행

## 6. 테스트 실행

```bash
# 전체 테스트 실행
./gradlew test

# 특정 테스트 클래스 실행
./gradlew test --tests HomeControllerTest

# 테스트 커버리지 확인
./gradlew test jacocoTestReport
```

## 7. 체크리스트

### 코드 작성 시

- [ ] 모든 테스트가 통과하는가?
- [ ] 매직 넘버/문자열을 상수로 추출했는가?
- [ ] 중복 코드를 제거했는가?
- [ ] 메서드명이 명확한가?
- [ ] 불필요한 객체 생성을 피했는가?

### Git 작업 시
- [ ] 모든 변경사항은 git add까지만 완료했는가?
- [ ] commit은 사용자가 명시적으로 요청했는가?
- [ ] staging 상태를 확인했는가? (git status)
- [ ] 불필요한 파일이 포함되지 않았는가?

### 커밋 시 (사용자 요청 시에만)
- [ ] 테스트가 모두 통과하는가?
- [ ] 코드 리뷰를 받았는가?
- [ ] 커밋 메시지가 명확한가?

## 8. 스킬 생성 규칙

> **새 스킬 생성 시 반드시 `/skill-creator` 스킬을 참조할 것**

### 필수 참조 파일

```
~/.claude/skills/util/skill-creator/SKILL.md
```

### 스킬 생성 워크플로우

1. **skill-creator 스킬 읽기** (필수)
2. 카테고리 선택 (write/export/data/util)
3. SKILL.md 작성 (템플릿 준수)
4. 프로젝트 CLAUDE.md 업데이트
5. git add

### 체크리스트

- [ ] skill-creator 스킬을 참조했는가?
- [ ] 메타데이터 형식을 준수했는가? (name, description)
- [ ] 500줄 이하인가?
- [ ] 프로젝트 문서에 등록했는가?
